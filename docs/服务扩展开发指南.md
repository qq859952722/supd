## 1. 项目概览

supd 是一个面向家庭 NAS 的进程监督器，用 Go 编写，前端用 React + TypeScript。

**核心能力**：
- 管理服务进程（启动/停止/重启/信号）
- 扩展系统（脚本任务，支持多种触发器）
- Web UI（文件编辑器、服务管理、扩展管理）
- 热重载（配置变更自动生效）
- 长轮询事件推送

**技术约束**：
- 无数据库（纯内存 + 文件）
- 无 SSE/WebSocket（长轮询）
- 单二进制部署（前端嵌入 Go 二进制）

## 3. 服务开发

### 3.1 service.yaml 完整字段

源码定义：[internal/config/service.go](file:///home/qq/Documents/trae_projects/supd/internal/config/service.go)

```yaml
# === 基本信息 ===
name: qbittorrent              # 服务名（唯一标识，必填）
version: "5.2.3"               # 服务版本号
description: "qBittorrent client" # 服务描述
icon: database                 # 图标名（前端 IconPicker 中选）
autostart: true                # supd 启动时是否自动启动（默认 false）

# === 启动命令 ===
command:                       # 启动命令（数组，必填）
  - ./bin/qbittorrent-nox
  - --confirm-legal-notice
  - --webui-port=9091
  - --profile=./data/qbittorrent
runtime: ""                    # 运行时别名（如 bash/python3），为空则直接执行 command[0]
user: ""                       # 以指定用户运行（需 root 权限）
group: ""                      # 以指定组运行
workdir: ""                    # 工作目录（空则使用服务所在目录）
depends_on: []                 # 依赖服务列表（仅做拓扑排序提示，不强制自动启动依赖）
tags: [download, media]        # 标签（用于分组和筛选）

# === Readiness 检查 ===
readiness:
  type: tcp_check              # 4种类型：fd_notify/tcp_check/http_check/script
  port: 9091                   # tcp_check: 检查的端口
  interval_seconds: 1          # 检查间隔（默认 1）
  timeout_seconds: 5           # 超时时间（默认 5）

# === 重启策略 ===
restart:                       # 可省略，使用 config.yaml 中 defaults.restart
  policy: always               # always | on-failure | never
  backoff_ms: 2000             # 初始退避毫秒
  max_backoff_ms: 60000        # 最大退避毫秒
  multiplier: 3                # 退避倍数
  max_retries: 5               # 最大重试次数
  reset_after_seconds: 600     # 成功运行多久后重置重试计数

# === 停止配置 ===
stop:
  grace_seconds: 10            # SIGTERM 后等待秒数（默认 10）
  timeout_seconds: 60          # 总超时秒数，超时后 SIGKILL（默认 60）

# === 日志配置 ===
logging:
  enabled: true                # 是否采集日志（默认 true）
  max_size_mb: 10              # 单文件最大 MB（默认 10）
  max_files: 5                 # 保留文件数（默认 5）

# === 自定义信号 ===
signals:
  reload: SIGHUP               # 重载信号
  rotate_logs: SIGUSR1         # 日志轮转信号
  graceful_quit: SIGQUIT       # 优雅退出信号

# === 打包配置 ===
package:
  include: [bin/, config.yaml] # 导出时包含的路径
  exclude: [data/]             # 导出时排除的路径
  default: include             # 默认行为：include | exclude
```

### 3.2 Readiness 检查的 4 种类型

| 类型 | 配置字段 | 说明 |
|---|---|---|
| `fd_notify` | `fd: 3` | 等待子进程通过 fd 发送就绪通知（supd 监听该 fd） |
| `tcp_check` | `port: 9091` | 周期性 TCP 连接检查端口是否可连 |
| `http_check` | `url: http://127.0.0.1:9091/api/v2/app/version`<br>`expected_status: 200` | HTTP GET 检查返回状态码 |
| `script` | `check: [./health-check.sh]` | 执行脚本，退出码 0 = 就绪 |

通用字段：
- `interval_seconds`：检查间隔（默认 1）
- `timeout_seconds`：总超时（默认 5），超时后服务状态变 `failed`

### 3.3 服务状态机（7 种状态）

```
pending → starting → up → ready
                ↓         ↓
              failed    stopping → down
                ↓
            (restart 或 manual clear)
```

| 状态 | 说明 |
|---|---|
| `pending` | 已加载配置但未启动 |
| `starting` | 进程已启动，等待 readiness 检查通过 |
| `up` | 进程运行中（无 readiness 配置时直接到此） |
| `ready` | readiness 检查通过 |
| `stopping` | 正在停止（执行 graceful shutdown） |
| `down` | 已正常停止 |
| `failed` | 异常退出或 readiness 超时 |

**关键规则**：
- 自动重启**不经过** `down` 状态（直接 `failed → starting`）
- `failed` 状态需手动 `clear-failed` 或等待自动重启
- `stopping` → `down` 不经过 `failed`

### 3.4 进程管理

**启动**：
- 子进程在独立进程组中（`Setpgid: true`），便于整组发信号
- PID 文件记录在 `<workdir>/.supd/pids/<service>.pid`，supd 异常退出后下次启动时清理孤儿进程

**停止流程**（graceful shutdown）：
1. 发送 `SIGTERM` 到进程组
2. 等待 `grace_seconds`（默认 10 秒）
3. 若仍未退出，发送 `SIGKILL`
4. 总超时 `timeout_seconds`（默认 60 秒）

**信号**：
- `POST /api/services/{name}/signal`：发送自定义信号
- `POST /api/services/{name}/force-stop`：直接 SIGKILL

### 3.5 热重载

- fsnotify 监听 `services/` 和 `extensions/` 目录
- **500ms 防抖**：文件变更后等 500ms 再处理，避免频繁触发
- **字段变更分类**：
  - 元数据变更（name/description/icon/tags）：即时热重载，不重启服务
  - 命令变更（command/runtime/workdir）：标记"待激活"，下次重启生效
  - readiness/restart/stop 变更：下次启动生效
- **热重载不影响运行中的服务**：配置变更不自动重启服务

### 3.6 端口收集

supd 自动收集服务进程占用的端口：
1. 读取 `/proc/net/{tcp,tcp6,udp,udp6}` 获取所有监听 socket
2. 读取 `/proc/<pid>/fd/*` 匹配 socket inode
3. 返回 `ResourceResponse.ports` 字段，包含协议（TCP/UDP）和地址

前端通过 `no-cors` fetch 探测 HTTP 端口（resolve 即为 HTTP）。

### 3.7 服务目录结构

```
services/<service-name>/
├── service.yaml          # 服务配置（必填）
├── env.yaml              # 服务级环境变量（可选，规格 §2.2.4）
├── bin/                  # 服务二进制（可选）
├── data/                 # 服务数据目录（可选）
└── extensions/           # 服务级扩展（可选）
    └── <ext-name>/
        ├── meta.yaml
        ├── env.yaml      # 扩展私有环境变量（可选）
        └── run.sh
```

> **注意**：环境变量文件统一命名为 `env.yaml`（不是 `.env`），格式为 `{value, enabled, hint}`，详见 §5.3。

## 4. 扩展开发

### 4.1 meta.yaml 完整字段

源码定义：[internal/config/extension.go](file:///home/qq/Documents/trae_projects/supd/internal/config/extension.go)

```yaml
# === 基本信息 ===
name: qbittorrent-health-check    # 扩展名（唯一标识，必填）
version: "1.0.0"                  # 扩展版本
description: "健康检查扩展"        # 描述
enabled: true                     # 是否启用（默认 true）

# 说明：服务关联由目录结构决定（services/<svc>/extensions/<ext>/），
# meta.yaml 中无需（也不解析）service 字段

# === 执行配置 ===
runtime: bash                     # 运行时别名（bash/python3 等），为空则直接执行
entry: /abs/path/run.sh           # 脚本入口路径（必填，建议用绝对路径）
timeout_seconds: 60               # 超时秒数（默认 600，硬上限 1800）
run_as: ""                        # 执行身份（空则继承服务 user 或当前用户）
concurrency: replace              # 并发策略（默认 replace）

# === UI 配置 ===
ui:
  show_logs: true                 # 是否在 UI 显示运行日志（默认 true）
  button_style: default           # 默认按钮样式：primary/default/danger

# === 触发器 ===
triggers:
  on_demand: true                 # 手动触发（有 actions 时默认 true）
  on_schedule:                    # 定时触发
    - cron: "*/5 * * * *"
      action: check
  service_lifecycle:              # 服务生命周期触发（服务级扩展）
    - event: post_ready
      action: check
    - event: on_failure
      action: diagnose
    - event: pre_stop
      action: cleanup
  supd_lifecycle:                 # supd 生命周期触发（全局扩展）
    - event: pre_start
      action: on-startup
    - event: post_ready
      action: on-ready
    - event: pre_shutdown
      action: on-shutdown

# === Action 定义 ===
actions:
  - id: check                     # action 唯一标识
    label: 健康检查                # 按钮显示文本
    button_style: primary         # 按钮样式：primary/default/danger
    args: [--verbose]             # 附加参数（追加到 entry 后）
    # 注意：规格 §2.2.3 还定义了 icon 和 enabled 字段，但当前实现未支持
    # 在 meta.yaml 中添加 icon 字段会触发 strictValidateByFileName 校验失败（DEV-012）
```

**Action 字段实现状态**：
- ✅ `id` / `label` / `button_style` / `args` — 已实现
- ❌ `icon` / `enabled` — 规格要求但未实现（DEV-012），添加会导致严格校验失败

### 4.2 触发器类型（4 种）

| 触发器 | 配置键 | 事件 | 适用 |
|---|---|---|---|
| `on_demand` | `on_demand: true` | 手动点击按钮 | 所有扩展 |
| `on_schedule` | `on_schedule: [{cron, action}]` | cron 定时 | 所有扩展 |
| `service_lifecycle` | `service_lifecycle: [{event, action}]` | `pre_start`/`post_ready`/`on_failure`/`pre_stop` | 服务级扩展 |
| `supd_lifecycle` | `supd_lifecycle: [{event, action}]` | `pre_start`/`post_ready`/`pre_shutdown` | 全局扩展 |

**cron 表达式**：标准 5 段式 `分 时 日 月 周`，如 `*/5 * * * *`（每 5 分钟）。

**service_lifecycle 事件**：
- `pre_start`：服务启动前（`SUPD_SERVICE_PID=0`）
- `post_ready`：服务就绪后（`SUPD_SERVICE_PID` 有值）
- `on_failure`：服务异常退出时（注入 `SUPD_SERVICE_EXIT_CODE`/`SUPD_SERVICE_SIGNAL`/`SUPD_SERVICE_RESTART_COUNT`）
- `pre_stop`：服务停止前

### 4.3 并发策略（4 种）

| 策略 | 说明 | 示例 |
|---|---|---|
| `replace` | 取消前任务，启动新任务（默认） | `concurrency: replace` |
| `serialize` | 排队，前任务完成后才跑新的 | `concurrency: serialize` |
| `parallel` | 并行跑，不限数量 | `concurrency: parallel` |
| `debounce:Ns` | trailing debounce，N 秒内无新触发才执行 | `concurrency: debounce:5s` |

**debounce 取值约束**：仅支持秒（`Ns`），**不支持毫秒格式**。N 取值范围 `[1, 3600]`（即 1 秒 ~ 1 小时）。源码：[internal/config/extension_validate.go](file:///home/qq/Documents/trae_projects/supd/internal/config/extension_validate.go) 的 `validateConcurrency`、[internal/extension/concurrency.go](file:///home/qq/Documents/trae_projects/supd/internal/extension/concurrency.go) 的 `ParseConcurrency`。

### 4.4 Action 与 button_style

每个 action 对应 UI 上的一个按钮。

`button_style` 三种值：
- `primary`：主要按钮（高亮）
- `default`：默认按钮
- `danger`：危险按钮（红色，用于删除/诊断等）

action 的 `args` 会追加到 entry 命令后：`<runtime> <entry> <args...>`。

### 4.5 超时与三层防线

1. **扩展超时**（`timeout_seconds`，默认 600 秒）：到时发 `SIGTERM`，等 5 秒
2. **硬上限**（`extension_hard_limit_seconds`，默认 1800 秒）：到时直接 `SIGKILL`
3. **用户取消**：`POST /api/extensions/runs/{runID}/cancel` → `SIGKILL` 进程组

### 4.6 扩展执行 11 步流程

源码：[internal/extension/executor.go](file:///home/qq/Documents/trae_projects/supd/internal/extension/executor.go)

1. 生成 `run_id`（UUID）
2. 构造环境变量：`os.Environ()` + 合并层级 env + `SUPD_*` 上下文变量
3. 构造命令：`<runtime 路径> <entry> <args...>`
4. 设置进程组（`Setpgid: true`），关闭 stdin
5. 设置执行身份（`run_as` 解析）
6. 启动进程
7. 启动 stdout goroutine 解析协议（`::progress::` / `::result::`）
8. 启动 stderr goroutine 写日志
9. 启动超时定时器
10. 阻塞等待 `cmd.Wait()` 返回
11. 判定最终状态（优先级：timeout/killed/canceled > `::result::` 协议 > exit code）

### 4.7 服务级扩展 vs 全局扩展

| 特性 | 服务级扩展 | 全局扩展 |
|---|---|---|
| 目录 | `services/<svc>/extensions/<ext>/` | `extensions/<ext>/` |
| `service` 字段 | 不解析（服务关联由目录结构决定）| 不解析 |
| `service_lifecycle` | 支持 | 不支持 |
| `supd_lifecycle` | 不支持 | 支持 |
| 工作目录 | `filepath.Dir(meta.yaml)`（扩展自身目录）| `filepath.Dir(meta.yaml)` |
| `SUPD_SERVICE_DIR` | 注入（服务目录绝对路径） | 不注入 |
| `run_as` 默认值 | 服务的 `user`（DEV-005：仅显式 run_as 生效，默认值未完整接入）| 当前用户 |
| `env.yaml` 层级 | 全局 env + 服务 env + 扩展私有 env | 全局 env + 扩展私有 env |

### 4.8 扩展日志

- **存储位置**：`<log_dir>/extensions/<ext_name>/<run_id>.log`
- **服务级扩展**：双 writer — 同时写扩展日志和服务日志
- **上限**：10MB 硬编码（`MaxExtensionLogSize`）
- **查看 API**：`GET /api/extensions/runs/{runID}/logs`

### 4.9 扩展运行历史

- **保留**：7 天（内存，不持久化）
- **RunResult 字段**：run_id / extension_name / action_id / state / exit_code / progress / result_msg / result_level / started_at / finished_at / trigger_type / service_name
- **过滤查询**：`GET /api/extensions/runs?extension_name=xxx&state=failed`

## 5. 扩展脚本开发

### 5.1 脚本接口

扩展脚本可以是任意可执行文件（bash/python/go binary 等），通过 `runtime` + `entry` 指定执行方式。

**输入**：
- 环境变量（`SUPD_*` 系列，见下表）
- 命令行参数（action 的 `args` 追加到 entry 后）
- stdin：**已关闭**（不支持 stdin 输入）

**输出**：
- stdout：普通日志 + 协议指令（`::progress::` / `::result::`）
- stderr：全部按普通日志处理

**退出码**：
- `0` = success（若无 `::result::` 协议输出）
- 非 `0` = failed（若无 `::result::` 协议输出）
- `::result::` 协议优先于退出码

### 5.2 环境变量（SUPD_* 系列）

源码：[internal/extension/run_context.go](file:///home/qq/Documents/trae_projects/supd/internal/extension/run_context.go)

| 变量 | 适用场景 | 说明 |
|---|---|---|
| `SUPD_EVENT` | 所有扩展 | 事件类型：`on_demand`/`on_schedule`/`service_lifecycle`/`supd_lifecycle` |
| `SUPD_TRIGGER_SOURCE` | 所有扩展 | 触发源：`webui`/`cli`/`schedule`/`service_lifecycle`/`supd_lifecycle` |
| `SUPD_TRIGGER_TIME` | 所有扩展 | 触发时间（RFC3339 格式） |
| `SUPD_TRIGGER_USER` | 所有扩展 | 触发者标识 |
| `SUPD_RUN_ID` | 所有扩展 | 本次运行的 UUID |
| `SUPD_EXTENSION_NAME` | 所有扩展 | 扩展名 |
| `SUPD_ACTION` | 所有扩展 | 当前 action id |
| `SUPD_PHASE` | lifecycle 触发 | 阶段：`pre_start`/`post_ready`/`on_failure`/`pre_stop`/`pre_shutdown` |
| `SUPD_SERVICE` | service_lifecycle | 服务名 |
| `SUPD_SERVICE_PID` | service_lifecycle | 服务进程 PID（`pre_start` 时为 0） |
| `SUPD_SERVICE_DIR` | 服务级扩展 | 服务目录绝对路径 |
| `SUPD_SERVICE_EXIT_CODE` | on_failure | 服务退出码 |
| `SUPD_SERVICE_SIGNAL` | on_failure | 导致退出的信号编号 |
| `SUPD_SERVICE_RESTART_COUNT` | on_failure | 服务重启次数 |

### 5.3 环境变量层级合并

> 规格 §2.2.4：环境变量文件统一命名为 `env.yaml`（不是 `.env`），格式为 `{value, enabled, hint}`。

环境变量按以下顺序合并（后者覆盖前者同名变量）：

1. 全局 env 文件（`<baseDir>/env/*.yaml`，按文件名字母序加载）
2. 全局扩展私有 env（`<baseDir>/extensions/<ext>/env.yaml`）
3. 服务 env（`<baseDir>/services/<svc>/env.yaml`）
4. 服务级扩展私有 env（`<baseDir>/services/<svc>/extensions/<ext>/env.yaml`）
5. `SUPD_*` 上下文变量（运行时注入，最高优先级）

**env.yaml 格式**：

```yaml
env:
  <VAR_NAME>:
    value: <值，永远是字符串>
    enabled: true              # 可选，默认 true；false 时不注入
    hint: <说明文字>           # 可选，传给前端展示
```

**关键规则**：
- `enabled: false` 的变量不注入扩展进程环境，但 value 保留在 env.yaml 中
- 同名变量后者覆盖前者（包括 enabled 状态）
- 密码字段启发式识别：变量名包含 `PASSWORD`/`PWD`/`SECRET`/`TOKEN`/`KEY` 时前端按 password 渲染

### 5.4 脚本模板

```bash
#!/bin/bash
# 扩展脚本模板
set -euo pipefail

# 读取 supd 注入的环境变量
ACTION="${SUPD_ACTION:-default}"
SERVICE_DIR="${SUPD_SERVICE_DIR:-$(pwd)}"
RUN_ID="${SUPD_RUN_ID:-unknown}"

echo "[$(date)] 开始执行: action=$ACTION run_id=$RUN_ID"

# 输出进度
echo "::progress:: 50 \"处理中...\""

# 执行业务逻辑
if do_something; then
    echo "::result:: success \"操作成功\""
else
    echo "::result:: error \"操作失败: $?\""
    exit 1
fi
```

## 6. stdout 协议

源码：[internal/extension/protocol.go](file:///home/qq/Documents/trae_projects/supd/internal/extension/protocol.go)

扩展通过 stdout 输出约定格式的行，supd 实时解析。**stderr 不解析协议**。

### 6.1 进度指令

```
::progress:: <0-100> "<message>"
```

示例：
```bash
echo "::progress:: 30 \"下载中...\""
echo "::progress:: 60 \"安装中...\""
echo "::progress:: 100 \"完成\""
```

- 百分比范围 0-100（超出按普通日志处理）
- message 用双引号包裹（支持转义 `\"`）
- 多次输出取最后一次

### 6.2 结果指令

```
::result:: <success|warning|error> "<message>"
```

示例：
```bash
echo "::result:: success \"全部完成\""
echo "::result:: warning \"完成但磁盘空间不足\""
echo "::result:: error \"下载失败: 网络超时\""
```

- `success`：任务成功
- `warning`：视为成功（不计入失败统计），但显示警告
- `error`：任务失败
- `::result::` 协议**优先于** exit code

### 6.3 状态判定优先级

```
timeout / killed / canceled  >  ::result:: 协议  >  exit code
```

- 若进程被超时杀死 → `TaskTimeout`（忽略 `::result::`）
- 若进程被 SIGKILL → `TaskKilled`
- 若进程被用户取消 → `TaskCanceled`
- 否则按 `::result::` 协议判定
- 无 `::result::` 输出时按 exit code：0=success，非0=failed

### 6.4 行长限制

超过 8KB 的行按普通日志处理（不解析协议）。

### 7.7 请求/响应格式约定（API 格式避坑）

> 以下约定来自阶段三 API 自动化测试中实测验证的结果，避免常见请求格式错误。

#### 服务创建/更新

- `POST /api/services`、`PUT /api/services/{name}`：body 是 JSON，**`version` 字段必填**（虽然 `omitempty` 但 `Validate` 强制要求）
- 源码：[internal/api/service_handler.go](file:///home/qq/Documents/trae_projects/supd/internal/api/service_handler.go) 的 `CreateServiceRequest.Validate`

#### 服务配置文件更新

- `PUT /api/services/{name}/config`：body 是 JSON `{"content":"<yaml 字符串>"}`，**不是 raw YAML**
- 成功返回 `200`，配置内容写入 `service.yaml`
- 源码：[internal/api/file_handler.go](file:///home/qq/Documents/trae_projects/supd/internal/api/file_handler.go) 的 `handleWriteFile`

#### 文件操作（`/api/files/*`）

- **path 参数**：放在 **URL query**（`?path=<abs>`），**不在 JSON body**
- **写文件**（`PUT /api/files/write`）：body 是 JSON `{"content":"<文本内容>"}`
- **移动文件**（`POST /api/files/move`）：body 是 JSON `{"destination":"<目标路径>"}`（字段名是 `destination`，不是 `dst`/`target`/`dest`）
- **上传文件**（`POST /api/files/upload`）：`multipart/form-data`，文件字段名为 `file`，目标路径通过 `?path=<abs>` query 传递，100MB 大小限制
- **成功响应**：写文件成功返回 `204 No Content`（不是 200）
- 源码：[internal/api/file_handler.go](file:///home/qq/Documents/trae_projects/supd/internal/api/file_handler.go)

#### 扩展创建

- `POST /api/extensions`：body 是 JSON，**`ExtensionMeta` 必填字段**：`name`/`version`/`entry`/`concurrency`/`enabled`/`triggers`
- **扩展名正则**：`^[a-z][a-z0-9-]*$`（小写字母开头，只允许小写字母/数字/连字符，**不允许下划线**）
- 源码：[internal/config/extension.go](file:///home/qq/Documents/trae_projects/supd/internal/config/extension.go) 的 `ExtensionMeta`

#### 扩展运行触发

- `POST /api/extensions/{name}/run`：扩展创建后需 **sleep 1s** 等待 `extProvider` 异步注册完成，立即触发会返回空 `run_id`
- 源码：[internal/api/service_ops.go](file:///home/qq/Documents/trae_projects/supd/internal/api/service_ops.go)

#### 长轮询事件流

- `GET /api/system/events`：`since` 参数支持 **RFC3339 时间字符串** 或 **数值纳秒时间戳**
- `wait` 参数：1-60 秒，控制服务器阻塞等待时长；超时返回空数组 + 当前时间游标
- **测试避坑**：若 `since` 是过去的时间戳，会立即返回历史事件而非阻塞
- 源码：[internal/api/longpoll.go](file:///home/qq/Documents/trae_projects/supd/internal/api/longpoll.go) 第 57-78 行

#### 依赖管理简化原则

- `handleStartService` **不自动启动依赖服务**，仅做拓扑排序提示
- 用户手动启动 `child` 服务时，若 `base` 依赖未启动，`child` 仍会尝试启动（不强制限制）
- 符合规格 §2.1.2 第 228 行："依赖管理只做提示，不强制限制启动和安装"
- 源码：[internal/api/service_ops.go](file:///home/qq/Documents/trae_projects/supd/internal/api/service_ops.go) 第 29-68 行的 `handleStartService`

#### 成功响应

- 多数成功响应返回 `200 OK` + JSON body
- **写文件**、**清除运行记录**、**删除运行日志**等无返回体的操作返回 `204 No Content`
- 创建类操作返回 `201 Created`

```bash
echo "::progress:: 30 \"压缩数据...\""
tar -czf "$BACKUP_DIR/data-$(date +%Y%m%d).tar.gz" -C "$SERVICE_DIR/data" .

echo "::progress:: 80 \"验证备份...\""
if [ -f "$BACKUP_DIR/data-$(date +%Y%m%d).tar.gz" ]; then
    echo "::result:: success \"备份完成\""
else
    echo "::result:: error \"备份文件不存在\""
    exit 1
fi
```

4. `chmod +x run.sh`
5. 在 Web UI 点击"执行备份"按钮运行

### 12.3 创建一个 cron 定时扩展

```yaml
# meta.yaml
name: scheduled-ping
version: "1.0.0"
enabled: true
runtime: bash
entry: /abs/path/extensions/scheduled-ping/run.sh
timeout_seconds: 10
triggers:
  on_schedule:
    - cron: "*/5 * * * *"
      action: ping
actions:
  - id: ping
    label: Ping
```

```bash
#!/bin/bash
# run.sh
echo "[ping] $(date) - cron 触发"
echo "::result:: success \"ping 完成\""
```

### 12.4 创建一个服务生命周期 hook

```yaml
# meta.yaml
name: auto-config
version: "1.0.0"
enabled: true
runtime: bash
entry: /abs/path/services/myapp/extensions/auto-config/run.sh
timeout_seconds: 30
triggers:
  service_lifecycle:
    - event: pre_start
      action: prepare
    - event: post_ready
      action: notify
    - event: on_failure
      action: diagnose
actions:
  - id: prepare
    label: 准备配置
  - id: notify
    label: 通知就绪
  - id: diagnose
    label: 诊断故障
```

```bash
#!/bin/bash
# run.sh
PHASE="${SUPD_PHASE}"
SERVICE="${SUPD_SERVICE}"

case "$PHASE" in
    pre_start)
        echo "[$SERVICE] 启动前准备..."
        ;;
    post_ready)
        echo "[$SERVICE] 已就绪，发送通知"
        ;;
    on_failure)
        EXIT_CODE="${SUPD_SERVICE_EXIT_CODE}"
        echo "[$SERVICE] 异常退出: code=$EXIT_CODE"
        echo "::result:: warning \"服务异常: exit=$EXIT_CODE\""
        ;;
esac
```

### 12.5 导出与导入

```bash
# 导出服务
supd export myapp -o myapp.tar.gz
# 或通过 API（浏览器下载）
curl -o myapp.tar.gz http://localhost:8080/api/services/myapp/export

# 导入服务（双次上传无状态模式）
supd import myapp.tar.gz
# 或通过 API 两步确认：
# 步骤 1：上传包，返回差异预览（archive_version/local_version/exists_local）
curl -X POST -F "archive=@myapp.tar.gz" http://localhost:8080/api/services/import
# 步骤 2：用户确认后，再次上传同一包执行实际导入（备份现有目录 → 解压覆盖 → 失败自动回滚）
curl -X POST -F "archive=@myapp.tar.gz" http://localhost:8080/api/services/import/confirm
```

> **注意**：导入是无状态双次上传模式，无 session_id。两步都上传完整包。包内顶层目录会被自动剥离（如 `myapp-v1.0.0/service.yaml` → `service.yaml`）。

## 附录：开发工作流

### 新增服务流程

1. 创建 `services/<name>/` 目录
2. 编写 `service.yaml`
3. 放入二进制或脚本
4. （可选）添加 `env.yaml` 环境变量文件（规格 §2.2.4）
5. `supd validate services/<name>/service.yaml` 校验配置
6. 启动 supd，服务自动发现（fsnotify）或 `POST /api/reload`

### 新增扩展流程

1. 确定是服务级还是全局扩展
2. 创建对应目录 `services/<svc>/extensions/<ext>/` 或 `extensions/<ext>/`
3. 编写 `meta.yaml`
4. 编写 `run.sh`（或其他脚本）
5. `chmod +x run.sh`
6. （可选）添加 `env.yaml` 环境变量文件（规格 §2.2.4）
7. supd 自动发现，或在 Web UI 创建

### 修改已有服务/扩展

#### 直接编辑 + 热重载

1. 直接编辑 `service.yaml` 或 `meta.yaml`
2. 保存后 fsnotify 自动触发热重载（500ms 防抖）
3. 元数据变更即时生效；命令变更需重启服务
4. 或通过 API：`PUT /api/services/{name}` 或 `PUT /api/extensions/{name}`

#### 为已有服务添加扩展

**目标**：服务已存在并运行，需要为其新增扩展而不重启服务。

1. **选择位置**（优先服务级，便于随服务打包）：
   - 服务级：`services/<svc>/extensions/<ext-name>/`
   - 全局：`extensions/<ext-name>/`
2. **创建目录与文件**：`meta.yaml` + `run.sh`（`chmod +x`）
3. **meta.yaml 关键字段**：服务关联由目录结构决定，meta.yaml 中无需 `service` 字段（该字段不被解析）
4. **编写 run.sh**：通过 `SUPD_SERVICE`/`SUPD_SERVICE_PID`/`SUPD_SERVICE_DIR` 获取服务上下文；**不要硬编码端口**，从 service.yaml 或环境变量动态读取
5. **热重载自动加载**：新增扩展目录被 fsnotify 检测，**无需重启 supd 或服务**
   - `on_demand`：立即可触发
   - `on_schedule`：按 cron 下次触发
   - `service_lifecycle`：下次服务生命周期事件触发
6. **验证**：`GET /api/services/<svc>/extensions` 应返回新扩展；`POST /api/extensions/<ext-name>/run` 手动触发测试

#### 修改已有扩展（热重载行为对照）

| 修改对象 | 生效方式 | 影响运行中任务 |
|---|---|---|
| meta.yaml 元数据（description/ui/icon） | 热重载即时生效 | 否 |
| triggers 配置 | 热重载即时生效 | 否（已运行任务继续） |
| actions 定义 | 热重载即时生效 | 否 |
| concurrency 策略 | 热重载即时生效 | 否（已运行任务按旧策略完成） |
| timeout_seconds | 热重载即时生效 | 否（已运行任务按旧超时） |
| enabled（禁用/启用） | 热重载即时生效 | 否（已运行任务继续） |
| run_as | 热重载，下次执行生效 | 否 |
| run.sh 脚本内容 | **下次执行时生效** | 否（已运行任务用旧脚本） |
| entry 路径 / runtime | 热重载，下次执行生效 | 否 |

**常见修改场景**：
- **调整触发器**（如 on_demand → on_schedule）：改 triggers，热重载即时生效；检查 actions 中有对应 action id；改为 service_lifecycle 需确保扩展位于 `services/<svc>/extensions/` 目录下（服务关联由目录决定，无需 meta.yaml 字段）
- **新增/修改 actions**：热重载后 UI 自动刷新出现新按钮；id 不能重复；button_style 在 primary/default/danger 内
- **修改 run.sh 逻辑**：下次执行生效；已运行任务不受影响；修改后建议手动触发验证
- **临时禁用**：`enabled: false`，热重载即时生效；禁用后不触发（含 cron 和生命周期事件）；已运行任务执行完毕
- **调整超时**：`timeout_seconds` ≤ 1800，仅影响新任务
- **修改并发策略**：`concurrency` 格式正确（`debounce:Ns`，N∈[1,3600]，**不支持 Nms**），仅影响新任务
- **修改 run_as**：root/service/none，下次执行生效
- **升级版本号**：打包分发前必须更新，避免覆盖旧包
- **修改 entry 路径**：确保新脚本有执行权限；新增依赖文件放在扩展目录内随扩展打包

#### 修改已有服务配置（热重载行为对照）

| 修改对象 | 生效方式 | 影响运行中服务 |
|---|---|---|
| 元数据（description/icon/tags） | 热重载即时生效 | 否 |
| depends_on | 热重载即时生效 | 否 |
| readiness 配置 | 热重载，**下次启动生效** | 否（运行中服务不重新检测） |
| command/args/runtime | 热重载，**标记"待重启"** | 否（下次 start 生效） |
| stop 策略 | 热重载，下次停止生效 | 否 |
| logging 配置 | 热重载即时生效 | 否 |
| user/group | 热重载，**下次启动生效** | 否 |
| signals | 热重载，下次信号生效 | 否 |
| autostart | 热重载即时生效 | 否（影响下次 supd 启动） |

**关键约束**：热重载**不会自动重启运行中的服务**，命令类变更标记为"待重启"，需用户手动 stop→start 或 restart 生效。

**常见修改场景**：
- **修改服务端口**：同步改 service.yaml readiness.port + **所有引用该端口的扩展脚本**（最易遗漏）
- **修改启动命令**：改 command + args，检查 signals，重启服务生效
- **修改依赖关系**：改 depends_on，即时生效，但不会自动重启已启动的服务
- **修改停止策略**：改 stop.grace_seconds（默认 10）/ stop.timeout_seconds（默认 60），下次停止生效
- **修改重启策略**：改 restart.policy / restart.delay / restart.max_attempts

### 打包发布

1. `supd export <name>` 生成 tar.gz
2. 分发 tar.gz 文件
3. 目标机器 `supd import <file>.tar.gz`
4. 确认差异后完成导入
