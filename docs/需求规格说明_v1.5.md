## 第一部分：项目定位与总体设计

### 1.1 项目目标

构建一个用于家庭 NAS 场景的进程监督器，作为 PID 1 或系统服务管理 NAS 上有限的守护进程（< 20 个）。学习 S6（skarnet.org 的 supervision suite）的核心思想——监督树、自动重启、readiness 通知、per-service logging——但不兼容 S6 的任何二进制或配置格式。提供人性化、美观、傻瓜式的 WebUI，支持简易脚本扩展（全局扩展和服务级扩展），支持定时任务。

### 1.2 应用场景约束

- **使用者**：家庭 NAS 单用户，技术爱好者
- **网络**：内网环境，HTTP 即可，不做 HTTPS
- **规模**：管理 < 20 个服务，< 50 个扩展
- **部署**：单二进制文件，systemd 管理
- **性能基线**：在 20 服务 + 50 扩展满载场景下，supd 自身常驻内存 < 50MB（不含日志缓冲），CPU 空闲时 < 1%，单次 API 响应 < 100ms（不含长轮询挂起时间），长轮询并发上限 50 个连接（单客户端最多 5 个），超限返回 503 + SERVICE_BUSY
- **性能**：不追求高并发，资源占用低
- **可靠性**：家庭场景，允许 supd 重启后丢失运行时状态

### 1.3 设计原则

1. **简单克制**：避免过度设计，每个引入的特性必须有明确必要
2. **按需采集**：CPU/内存/进程等监控数据仅在前端请求时采集，后端不主动周期采样
3. **文件发现**：服务、扩展、运行时通过目录扫描自动发现，不集中注册
4. **扩展即一等公民**：hook、cron、手动触发统一为"扩展"抽象，触发时机不同而已
5. **不持久化运行时状态**：supd 重启后所有运行任务清空，仅日志和事件流文件保留
6. **HTTP 一统**：不使用 SSE/WebSocket，实时数据用长轮询
7. **学习不兼容**：借鉴 S6 思想，配置格式和二进制完全自研

### 1.4 总体架构

单二进制，单进程，goroutine 分工：

```
supd (单二进制)
├── 监督核心（goroutine）
│   ├── 服务依赖图
│   ├── 状态机
│   ├── 进程管理（fork+exec + 进程组）
│   └── 每服务一个 logger goroutine
├── 扩展系统（goroutine）
│   ├── 触发器调度（cron + lifecycle）
│   ├── 扩展执行器
│   └── 任务管理（内存）
├── 文件监听（fsnotify goroutine）
├── HTTP 服务器（goroutine）
│   ├── REST API
│   ├── 长轮询端点
│   └── 静态前端（embed.FS）
└── 优雅退出协调器
```

### 1.4.1 goroutine 通信规范

**通信方式**：

- goroutine 间通信统一用 Go channel，禁止共享状态 + mutex（仅 logger 的文件写互斥除外）
- 状态变更通过专属 channel 传递给状态机模块，状态机模块是唯一的状态写入者
- 禁止引入事件总线、消息队列、actor 模式、发布订阅模式

**包间依赖方向**：

```
cli → api → core/extension → config/storage/watch/logging/system/archive
```

- 单向依赖，禁止循环依赖
- 禁止引入 interface 抽象层解耦包间依赖（本项目无多实现需求）
- 禁止引入依赖注入容器（如 wire/fx）

### 1.5 三层抽象

1. **监督核心层**：进程管理、依赖图、状态机、日志收集
2. **扩展系统层**：触发器、执行器、环境变量合并、stdout 协议解析
3. **WebUI 层**：顶部导航 + 单页路由 + Monaco 编辑器 + 表单引擎

### 1.6 目录结构

### 1.7 关键概念定义

| 概念 | 定义 |
|---|---|
| **服务（Service）** | 一个被监督的 longrun 守护进程，由 service.yaml 定义 |
| **扩展（Extension）** | 一段可触发执行的脚本，统一抽象 hook、cron、手动触发 |
| **Action** | 扩展的一个执行入口，对应一个按钮，通过 CLI args 区分 |
| **触发器（Trigger）** | 扩展被触发的方式：on_demand / on_schedule / service_lifecycle / supd_lifecycle |
| **任务（Run）** | 扩展的一次具体执行，有唯一 run_id |
| **Readiness** | 服务从"进程已启"到"可提供服务"的过渡状态 |
| **环境变量层级** | 全局 env → 全局扩展私有 env → 服务 env → 服务级扩展私有 env，后者覆盖前者同名变量 |

### 1.8 时间与编码约定

- **ISO8601 时间**：所有时间戳必须带时区，格式如 `2026-07-07T15:30:00+08:00`；禁止不带时区或用 UTC（Z 后缀）
- **内部存储**：长轮询的 since 游标基于单调时钟（monotonic clock）的纳秒，避免 NTP 跳变影响；返回给前端时转为 ISO8601 带时区
- **字节单位**：所有字节大小字段用 MB（int 类型），禁止用 KB / GB / 字节
- **时间字段单位**：统一为 int 秒（字段名加 `_seconds` 后缀），唯一例外是 `backoff_ms` 保持毫秒（退避通常小于 1 秒）
- **cron 时区**：按服务器时区（Asia/Shanghai）执行，WebUI 在定时任务页面显示"按服务器时区执行"提示
- **字符编码**：UTF-8 假设，非 UTF-8 内容按字节流展示（可能乱码但不报错）

## 第二部分：详细需求设计

### 2.1 服务监督核心

**服务类型说明**：supd 仅支持 longrun 服务（长期运行的守护进程）。不支持 oneshot（一次性任务），一次性任务由扩展系统承担（通过 on_demand 或 on_schedule 触发）。

#### 2.1.1 服务状态机

服务必须遵循以下 7 个状态：

| 状态 | 含义 |
|---|---|
| `pending` | 已加载但依赖未就绪，等待中 |
| `starting` | 进程已启动，等待 readiness 信号 |
| `up` | 进程运行中。有 readiness 配置时为中间态（等待通过检查后转 ready）；无 readiness 配置时为终态 |
| `ready` | 通过 readiness 检查，可提供服务 |
| `stopping` | 已发送停止信号，等待进程退出 |
| `down` | 进程已退出，未自动重启 |
| `failed` | 永久失败，不再自动重启（达到 max_retries 或 policy=never） |

状态转移规则：

- `pending` → `starting`：所有 depends_on 服务进入 `ready` 后
- `starting` → `up`：进程启动后立即进入（无论是否有 readiness 配置）
- `up` → `ready`：readiness 检查通过（无 readiness 配置时 `starting → up` 即为终态，不经过 ready）
- `up` / `ready` → `stopping`：用户停止或重启
- `up` / `ready` → `starting`：进程死亡且 restart.policy 允许，经退避等待后直接重新 starting（不经过 down 状态）
- `up` / `ready` → `failed`：进程死亡且达到 max_retries，或 readiness 检查超时
- `starting` → `starting`：starting 阶段进程退出且 restart.policy 允许，经退避等待后重新 starting
- `starting` → `failed`：starting 阶段进程退出且 restart.policy 不允许重启（policy=never 或达到 max_retries），或 readiness 检查超时
- `stopping` → `down`：进程退出
- `down` / `failed` → `starting`：用户手动启动

**up 与 ready 的语义**：

- `up` 是"进程在跑但未通过 readiness"的中间态，仅表示进程在跑，不表示可接收请求
- `ready` 是"通过 readiness"的终态，表示可接收请求
- 无 readiness 配置的服务，`starting → up` 即为终态（不经过 ready）

**on-failure 触发条件**：

- 进程退出且退出码非 0，或进程被信号杀死（非 SIGTERM/SIGINT 的手动停止）
- 与服务处于 up 还是 ready 状态无关
- 手动停止（SIGTERM/SIGINT）不算 failure，不触发 on-failure 重启
- readiness 检查超时进入 failed 后，不再自动重启（需用户手动启动）

#### 2.1.2 依赖图

- 服务通过 `depends_on` 字段声明依赖
- 启动时按依赖图拓扑排序，按层级并行启动
- 同层服务并行启动，等待全部 ready 后再启动下一层
- 停止时按依赖反序停止（被依赖的先停）
- **循环依赖检测**：发现循环拒绝启动相关服务并在 WebUI 提示循环路径
- **自引用检测**：服务 A 的 depends_on 包含 A 自己，视为循环依赖，拒绝启动该服务
- **依赖服务不存在**：当前服务停留 pending 状态，WebUI 显示"依赖服务 X 不存在"警告
  - 用户可手动强制启动（自行承担风险）
  - 不让服务盲目进入 starting 又因 readiness 失败转 failed（避免迂回失败）
- **依赖服务存在但 failed**：当前服务进入 pending 等待，不自动启动
  - 用户可手动强制启动，但需自行承担风险
- **依赖状态变化自动重算**：依赖服务从 failed 恢复为 ready、从 down 启动为 ready 时，自动触发依赖图重算，原本因依赖未就绪而 pending 的服务自动从 pending 转为 starting（无需用户干预）
- **简化原则**：依赖管理只做提示，不强制限制启动和安装

#### 2.1.3 进程管理

详细机制见 2.1.8 进程管理核心机制。要点：

- 使用 `os/exec` + `syscall` 实现 fork+exec
- 每个子进程设置独立进程组（`SysProcAttr.Setpgid = true`）
- 信号发送使用 `syscall.Kill(-pgid, sig)` 发到整个进程组
- 不引入 cgroup
- 服务 stdin 必须关闭（`cmd.Stdin = nil`）
- 禁止服务进程后台化（command 必须是前台运行）

#### 2.1.4 信号与停止流程

单个服务停止流程：

1. 状态变更为 `stopping`
2. 触发该服务的 `service_lifecycle:pre_stop` 扩展（按目录字母序串行，每个扩展独立超时控制）
3. 向进程组发送 SIGTERM
4. 等待 `stop.grace_seconds`（默认 10 秒）
5. 进程仍未退出 → 发送 SIGKILL
6. 进程退出后状态变更为 `down`
7. **手动停止不算 failure**，不触发 on_failure 扩展

**停止流程总时限**：`stop.timeout_seconds`（默认 60 秒）覆盖整个停止流程（含 pre_stop 扩展执行 + SIGTERM 等待）。超时后直接 SIGKILL 进程组，不再等待。

#### 2.1.5 重启策略

service.yaml 的 `restart` 字段：

| 字段 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| `policy` | enum | `always` | `always` / `on-failure` / `never` |
| `backoff_ms` | int | 1000 | 初始退避毫秒数 |
| `max_backoff_ms` | int | 30000 | 最大退避毫秒数 |
| `multiplier` | int | 2 | 指数退避倍数 |
| `max_retries` | int | 0 | 0=无限；正数=失败 N 次后变 failed |
| `reset_after_seconds` | int | 300 | 进入 up 或 ready 后 N 秒重置重启计数器（retries 归零 + 退避时间回到 backoff_ms）。无 readiness 的服务用 up 作为重置锚点 |

退避计算：`min(backoff_ms * multiplier^retries, max_backoff_ms)`

#### 2.1.6 Readiness 检查

四种类型，由 `readiness.type` 字段指定：

| 类型 | 实现 | 字段 |
|---|---|---|
| `fd_notify` | 父进程创建 pipe，通过 `cmd.ExtraFiles` 将写端传递给子进程作为指定 fd，子进程往该 fd 写一行换行符即视为就绪 | `fd`（默认 3） |
| `tcp_check` | `net.DialTimeout("tcp", "127.0.0.1:port", 1s)` | `port`, `interval_seconds`, `timeout_seconds` |
| `http_check` | HTTP GET 检查状态码 | `url`, `expected_status`, `interval_seconds`, `timeout_seconds` |
| `script` | 执行指定脚本，exit 0 视为就绪 | `check`（命令数组）, `interval_seconds`, `timeout_seconds` |

- 无 readiness 配置的服务，进程启动即视为 `up`（注意不是 ready）
- `interval_seconds` 默认 1 秒（每次检查的间隔）
- `timeout_seconds` 默认 5 秒（从 starting 起算的总时限，超时则服务进入 `failed`）
- 单次检查失败不算"服务死亡"，仅"未就绪"，按 interval 重试直至 timeout

#### 2.1.7 日志系统

**每服务一个 logger goroutine**：

- 父进程在 fork 前创建 pipe，子进程的 stdout/stderr 重定向到 pipe 写端
- 子进程退出后，父进程关闭自身持有的 pipe 写端，logger goroutine 收到 EOF 后退出
- logger goroutine 从 pipe 读端读取行，写入 `/var/log/supd/services/<svc>/current`
- 日志行格式：`[ISO8601 时间戳] [级别] 原始内容`（级别由启发式判定：含 ERROR/error → error，WARN/warn → warn，其余 info）

**轮转策略**：

- 单文件超过 `logging.max_size_mb`（默认 10MB）触发轮转
- `current` 重命名为 `@<ISO8601>.s`（如 `@2026-07-06T15-30-00.s`）
- 归档文件数超过 `logging.max_files`（默认 5）时删除最旧的
- 归档文件名 **必须**用 ISO8601 格式，文件名安全（去除冒号，用连字符）
- **同秒内多次轮转**：归档时间戳精确到秒，同一秒内多次轮转时追加序号（如 `@2026-07-06T15-30-00.s`、`@2026-07-06T15-30-00-1.s`、`@2026-07-06T15-30-00-2.s`）

**全局 catch-all 日志**：

- supd 自身日志写入 `/var/log/supd/supd.log`
- 同时写 stderr（systemd journal 可见）
- 未配置独立 logger 的服务输出走 catch-all

**扩展运行日志**：

- 服务级扩展的输出写入对应服务的日志目录，前缀 `[ext:<ext-name>]`
- 全局扩展（on_schedule / on_demand 触发）的输出写入 `/var/log/supd/extensions/<ext-name>/<run_id>.log`
- supd_lifecycle 触发的扩展输出写入 `/var/log/supd/supd.log`

**日志搜索**：

- 实时日志流：tail 当前 current 文件，长轮询方式
- 历史搜索：后端 grep 实现，返回匹配行 + 上下文
- 单次搜索最多返回 1000 行匹配
- **不建索引**（家庭场景日志量小）

#### 2.1.8 进程管理核心机制

本节是 supd 的核心基础，必须严格按此实现。

**fork+exec 机制**：

- 使用 `os/exec.Command` 构造子进程（runtime 路径 + entry + args）
- `cmd.Start()` 执行 fork+exec，子进程继承 supd 的环境变量
- `cmd.Wait()` 阻塞等待子进程退出，回收僵尸
- 每个子进程对应一个专属 goroutine 调用 Wait，Wait 返回即知退出态

**进程组管理**：

- 每个子进程必须设置独立进程组：`cmd.SysProcAttr = &syscall.SysProcAttr{Setpgid: true}`
- 子进程的 PID 等于其 PGID（进程组组长）
- 信号发送到整个进程组：`syscall.Kill(-pgid, sig)`（注意负号）
- 这样可以杀掉子进程派生的孙进程（前提是孙进程没有 setsid 脱离组）
- **不引入 cgroup**，接受孙进程可能脱离进程组的局限

**僵尸进程回收**：

- 子进程退出后内核保留其退出态，直到父进程调用 wait
- 父进程不 wait → 子进程变僵尸（Z 状态）
- supd 为每个 fork 的子进程启动一个专属 goroutine 调用 `cmd.Wait()`
- Wait 返回即完成回收，不产生僵尸
- **不需要全局 reaper**——因为每个子进程都有专属 Wait 调用

**孤儿进程处理**：

- supd 自身被 SIGKILL 杀死时，其子进程变孤儿
- Linux 机制：孤儿进程被 PID 1（init/systemd）收养，PID 1 会自动 reap 退出态
- 容器场景：supd 是 PID 1，supd 死则容器整体重启
- **不做额外处理**，依赖系统机制

**孙进程的局限**：

- 子进程可能再 fork 孙进程，孙进程的父进程是子进程
- supd 无法 wait 孙进程，子进程死后孙进程被 init 收养
- **应对**：用进程组发信号能杀到孙进程（除非孙进程 setsid 脱离组）
- service.yaml 文档中说明"禁止服务进程 daemonize"

**stdin 关闭**：

- `cmd.Stdin = nil`，不绑定 stdin
- 防止扩展脚本写了 `read input` 但忘了重定向，导致阻塞等待输入
- 关闭 stdin 后，`read` 立即返回 EOF，不会卡死

**进程退出判定**：

- `cmd.Wait()` 返回 nil → exit 0 → success
- `cmd.Wait()` 返回 `*exec.ExitError` → exit 非 0 或被信号杀死
- `ExitError.ExitCode()` 返回退出码（被信号杀死时返回 -1）
- `ExitError.Error()` 含信号名（如 "signal: segfault"）

**启动优化**：

- 同层依赖的服务并行启动（goroutine + sync.WaitGroup）
- 每个服务独立 readiness 检查，互不阻塞
- 同层全部 ready 后才启动下一层

**停止优化**：

- 按依赖反序停止
- 同层服务必须并行停止（不串行），同层全部停止后才停止下一层
- 每个服务独立的 grace 计时器

#### 2.1.9 PID 1 模式

**默认行为**：supd 默认以 PID 1 模式运行，承担 PID 1 的特殊职责。

**PID 1 模式的额外行为**：

- 显式注册 SIGTERM/SIGINT 处理（PID 1 默认忽略这些信号，不显式注册则 systemctl stop 或 docker stop 会失效）
- 调用 `PR_SET_CHILD_SUBREAPER`（Linux 系统调用）承担孤儿回收，孤儿进程退出后由 supd 调用 wait 回收，避免僵尸堆积
- 关机流程中不依赖 systemd，自己完成全部清理

**非 PID 1 模式**：

- 通过 `--no-pid1` 标志显式关闭 PID 1 模式
- 不调用 `PR_SET_CHILD_SUBREAPER`，依赖 systemd 回收孤儿
- 适用于 supd 作为 systemd 服务运行、由 systemd 管理生命周期的场景

**SIGHUP 信号**：

- supd 自身收到 SIGHUP 触发全量配置热重载（不重启 supd 进程），等同于"重新扫描所有配置文件并应用变更"
- 不影响运行中的服务（除非配置变更要求重启）
- **禁止**将 SIGHUP 解释为"重启 supd"

### 2.2 扩展系统

#### 2.2.1 扩展统一抽象

扩展是唯一的脚本执行单元，**取消独立的 hook、cron、task 概念**，全部统一为扩展的不同触发方式。

扩展的两种位置：

- **全局扩展**：`/etc/supd/extensions/<ext-name>/`
- **服务级扩展**：`/etc/supd/services/<svc>/extensions/<ext-name>/`

#### 2.2.2 扩展目录结构

```
<ext-name>/
├── meta.yaml        # 元数据：触发时机、actions、UI 声明
├── env.yaml         # 环境变量（{value, enabled, hint} 格式）
├── run.sh           # 入口脚本（或 run.ts / run.py 等，由 runtime 决定）
└── lib/             # 可选辅助文件
```

#### 2.2.3 meta.yaml 字段规范

```yaml
name: <扩展名>              # 必填，与目录名一致
version: <语义化版本>        # 必填，如 1.0.0
description: <描述>          # 可选
enabled: true                # 可选，默认 true

runtime: bash                 # 可选，运行时别名，默认 bash；引用三层来源（内置/config/runtimes扫描）
entry: run.sh                 # 必填，相对扩展目录的入口文件
timeout_seconds: 600          # 可选，默认 600（10 分钟）
run_as: service               # 可选，root | service | <用户名>；默认继承（User 模式，与 run_as_uid 互斥）
run_as_uid: 1000              # 可选，直接指定 uid（UID 模式，与 run_as 互斥）
run_as_gid: 1000              # 可选，直接指定 gid（0 或省略 = run_as_uid）
run_as_groups: [27, 100]      # 可选，补充组 gid 列表
concurrency: replace          # 可选，replace | serialize | parallel | debounce:5s；默认 replace

ui:                          # 可选
  icon: <图标名>             # 前端图标库 key（如 database、play）
  category: <分类>           # 前端分组（general/maintenance/monitoring/notify）

actions:                     # 可选，无则不在前端展示按钮，但扩展仍在列表中显示状态和历史
  - id: <唯一标识>           # 用作 CLI args
    label: <按钮文字>
    icon: <图标名>           # 可选
    button_style: default    # primary | default | danger
    args: [<CLI 参数>]       # 可选，省略或空数组时直接执行 entry（runtime entry）
    enabled: true            # 可选，默认 true

triggers:                    # 可选，无则仅手动触发
  - type: on_demand          # 手动触发，前端展示按钮
    action: <action id>      # 引用 actions 中的 id
    
  - type: on_schedule        # 定时触发
    schedule: <cron 表达式>  # 标准 5 段 cron
    action: <action id>
    retry_on_failure:        # 可选
      max_retries: 3
      interval_minutes: 5
    # retry_on_failure 说明：失败后每次重试生成新的 run_id，
    # 在任务历史中标记为重试，max_retries 用尽后不再重试
  
  - type: service_lifecycle  # 服务生命周期
    phase: [pre_start, post_ready, on_failure, pre_stop]  # 至少一个
    services: ["*"]          # "*" 或服务名列表
    action: <action id>
  
  - type: supd_lifecycle  # supd 生命周期
    phase: [pre_start, post_ready, pre_shutdown]  # 至少一个
    action: <action id>
```

**actions 的 args 字段调用规则**：

- 有 args（非空数组）：执行 `<runtime> entry arg1 arg2 ...`
- 无 args（省略或空数组）：执行 `<runtime> entry`（直接运行整个脚本）
- 脚本内部通过 `$1`（shell）、`process.argv[2]`（node）等区分调用方式
- triggers 中的 `action` 字段引用 actions 的 id，触发时使用对应 action 的 args


**triggers.action 引用不存在的 id**：

- 扩展加载失败，标记为"配置错误"状态
- 不影响其他扩展和服务
- WebUI 提示具体扩展名和引用的 action id

#### 2.2.4 环境变量层级合并

**合并顺序**（从低到高，后者覆盖前者同名变量）：

1. 全局 env 文件（`/etc/supd/env/*.yaml`，按文件名字母序加载，后者覆盖前者）
2. 全局扩展私有 env（`/etc/supd/extensions/<ext>/env.yaml`）
3. 服务 env（`/etc/supd/services/<svc>/env.yaml`）
4. 服务级扩展私有 env（`/etc/supd/services/<svc>/extensions/<ext>/env.yaml`）

每条环境变量格式：

```yaml
env:
  <VAR_NAME>:
    value: <值，永远是字符串>
    enabled: true              # 可选，默认 true；false 时不注入
    hint: <说明文字>           # 可选，传给前端展示
```

**关键规则**：

- `enabled: false` 的变量不注入扩展进程环境，但 value 保留在 env.yaml 中
- 同名变量后者覆盖前者（包括 enabled 状态）
- 密码字段启发式识别：变量名包含 `PASSWORD` / `PWD` / `SECRET` / `TOKEN` / `KEY` 时前端按 password 渲染

#### 2.2.5 扩展执行模型

**执行流程**：

1. 生成 run_id（UUID）
2. 构造环境变量：合并层级 env + 注入的 SUPD_* 上下文变量
3. 构造命令：`<runtime 路径> entry args...`（省略 runtime 时直接 `entry args...`）
4. 设置进程组（`Setpgid: true`），关闭 stdin
5. 设置执行身份（run_as 配置）
6. 启动进程，获取 stdout/stderr pipe
7. 启动 goroutine 解析 stdout 协议
8. 启动 goroutine 写日志（所有输出，含未识别的协议行）
9. 启动超时定时器（取 timeout_seconds 与全局硬上限较小值）
10. 阻塞等待 `cmd.Wait()` 返回
11. 判定最终状态，记录任务历史

**SUPD_* 上下文环境变量**（所有扩展运行时注入）：

| 变量名 | 说明 | 适用场景 |
|---|---|---|
| `SUPD_EVENT` | 事件类型：on_demand / on_schedule / service_lifecycle / supd_lifecycle | 所有 |
| `SUPD_TRIGGER_SOURCE` | 触发源：webui / cli / schedule / service_lifecycle / supd_lifecycle | 所有 |
| `SUPD_TRIGGER_TIME` | ISO8601 触发时间 | 所有 |
| `SUPD_TRIGGER_USER` | 触发者标识（记录用，非执行用户）。webui 触发固定为 "webui"；cli 触发为当前系统用户名；schedule 触发固定为 "schedule"；service_lifecycle 触发固定为 "service_lifecycle"；supd_lifecycle 触发固定为 "supd_lifecycle"。执行用户由 run_as 决定，与此字段无关 | 所有 |
| `SUPD_PHASE` | 阶段：pre_start / post_ready / on_failure / pre_stop | lifecycle 触发 |
| `SUPD_SERVICE` | 服务名 | service_lifecycle |
| `SUPD_SERVICE_PID` | 服务 PID | service_lifecycle |
| `SUPD_SERVICE_EXIT_CODE` | 服务退出码（on_failure 时） | on_failure |
| `SUPD_SERVICE_SIGNAL` | 服务信号（on_failure 时） | on_failure |
| `SUPD_SERVICE_RESTART_COUNT` | 服务当前重启次数 | on_failure |
| `SUPD_RUN_ID` | 本次运行的 run_id | 所有 |
| `SUPD_EXTENSION_NAME` | 扩展名 | 所有 |
| `SUPD_ACTION` | action id | 所有 |

**各 phase 下 SUPD_SERVICE_PID 的值**：

- `pre_start`：空字符串（服务进程尚未启动）
- `post_ready` / `pre_stop`：服务进程的实际 PID
- `on_failure`：进程退出前的 PID（进程已退出，PID 仅用于日志记录，可能已被系统回收）

**全局扩展在 service_lifecycle 触发时的 SUPD_SERVICE 值**：

- 注入触发该扩展的具体服务名（与触发源服务一致）

**各 phase 下 SUPD_SERVICE_EXIT_CODE / SUPD_SERVICE_SIGNAL 的值**：

- 仅 `on_failure` 阶段有实际值（来自 cmd.Wait 返回的 ExitError）
- 其他 phase 为空字符串或 `0`

#### 2.2.6 stdout 协议规范

扩展通过 stdout 输出约定格式的行，supd 实时解析。**stderr 全部按普通日志处理，不解析协议**。

协议指令（严格匹配，未识别行按普通日志处理）：

```
::progress:: <0-100> "<message>"
::result:: <success|warning|error> "<message>"
```

**字段说明**：

- `::progress::` 后接百分比（0-100 整数）和双引号包裹的消息（消息内双引号需转义为 `\"`）
- `::result::` 后接状态（success / warning / error）和双引号包裹的消息
- 一个 run 只能有一个 result，多次输出以最后一次为准
- 输出 result 后 supd 仍等进程退出才标记 run 结束
- 进程退出码与 result 状态不强绑定（exit 0 + result error 合法）

**解析容错**：

- 百分比非数字 → 按普通日志处理
- 状态非枚举值 → 按普通日志处理
- 行长超过 8KB → 截断，按普通日志处理
- 未识别的 `::xxx::` 前缀 → 按普通日志处理

#### 2.2.7 并发策略

每个扩展的 `concurrency` 字段控制同一 action 多次触发的行为：

| 策略 | 行为 |
|---|---|
| `replace`（默认） | 取消前任务（SIGTERM → 5 秒 → SIGKILL），启动新任务 |
| `serialize` | 排队，前任务完成后才跑新的 |
| `parallel` | 并行跑，不限数量 |
| `debounce:5s` | trailing debounce：每次触发重置 5 秒计时器，5 秒内无新触发后执行最后一次触发的 action。debounce 计时到期触发执行时，如有运行中的同 action 任务，按该扩展的 concurrency 策略处理（默认 replace） |

**多 action 之间互不阻塞**——每个 action 独立跟踪并发状态。

#### 2.2.8 超时与强杀

**三层防线**：

1. 扩展 meta.yaml 的 `timeout_seconds`（默认 600 秒，即 10 分钟）超时
2. 发送 SIGTERM 给整个进程组，等待 5 秒
3. 5 秒后仍未退出 → 发送 SIGKILL

**全局硬上限**：

- config.yaml 的 `extension_hard_limit_seconds`（默认 1800 秒，即 30 分钟）
- 无论 meta.yaml 怎么配，单次运行不能超过此上限
- 超过此上限强制 SIGKILL，并记录警告日志

#### 2.2.9 任务可靠性

- **核心原则**：无条件信任 `cmd.Wait()`——只要 Wait 返回，进程一定退出，状态一定可判定
- **不做心跳检测**：不监测"任务长时间无输出"等异常
- **不做状态持久化**：supd 重启后所有运行任务清空，前端检测到 supd 启动时间变化后清空本地状态
- **任务历史仅存内存**：保留 7 天，与日志文件同步清理

#### 2.2.10 任务状态枚举

| 状态 | 含义 |
|---|---|
| `pending` | 已创建但还没启动（如等待前任务取消） |
| `running` | 进程在跑 |
| `success` | exit 0 或 `::result:: success` 或 `::result:: warning` |
| `failed` | exit 非 0 或 `::result:: error` |
| `timeout` | 超时被杀 |
| `canceled` | 用户取消或被新任务 replace |
| `killed` | SIGKILL 强杀（取消时 5 秒不退） |

**warning 的归属**：`::result:: warning` 视为成功完成（带告警），不计入失败统计、不触发 on_failure 重试。warning 用于"完成但有非致命问题"的场景（如备份成功但磁盘空间不足）。

#### 2.2.11 失败处理

- 扩展运行失败（超时 / 退出码非 0 / result error）**仅记录日志，不影响服务**
- service_lifecycle 扩展失败 **不阻止服务启动或停止**（即使 pre_start 扩展失败，服务仍正常启动）。设计理由：扩展是辅助功能，不应让一个准备脚本阻塞核心服务
- 扩展输出写入对应日志目录，事件流记录"extension_failed"事件

#### 2.2.12 触发执行顺序与工作目录

**工作目录统一规则**：

所有扩展执行时的工作目录统一放在配置文件夹根目录的 `script_tmp/` 下，集中管理临时文件：

- 全局扩展工作目录：`/etc/supd/script_tmp/global+<ext-name>/`
- 服务级扩展工作目录：`/etc/supd/script_tmp/<svc-name>+<ext-name>/`

**设计说明**：

- 扩展脚本产生的临时文件集中在一个目录，便于用户手动清理
- 不会污染扩展自身目录或服务目录
- **程序不自动清理 script_tmp/**，由用户自主处理（可用扩展定时清理，如 on_schedule 触发的清理扩展）
- 扩展运行前程序不清空该目录，保留上次运行的临时文件便于调试

**触发执行顺序**：

同一时机多个扩展被触发时：

- 以服务为粒度独立执行：每个服务触发的扩展各自独立，不同服务的扩展可并行
- 同一服务内：先串行执行该服务匹配的全局扩展，再串行执行该服务的服务级扩展
- 同类型（全局或服务级）内部按目录名字母序串行执行（不并行）
- 每个扩展独立超时控制
- 前一个失败不影响后一个执行

#### 2.2.13 run_as 字段语义

执行身份支持两种互斥的配置模式：

- **User 模式**（按用户名）：通过用户名查找 uid/gid，需目标用户存在于 `/etc/passwd`
  - 服务：`user`（+ 可选 `group` 覆盖主组 gid，保留补充组）
  - 扩展：`run_as`
- **UID 模式**（按数字）：直接指定 uid/gid/groups，不查 `/etc/passwd`，适用于用户不在系统用户列表的场景（如 NAS 固定 uid 服务）
  - 服务：`uid` / `gid` / `groups`
  - 扩展：`run_as_uid` / `run_as_gid` / `run_as_groups`

**互斥规则**：User 模式与 UID 模式互斥，同时指定则配置校验报错（如服务同时配置 `user` 和 `uid`，或扩展同时配置 `run_as` 和 `run_as_uid`）。`gid`/`run_as_gid` 省略（=0）时取 `uid`/`run_as_uid` 的值。

**默认继承规则**：

- **全局扩展**：默认身份 = supd 启动用户
  - 显式 `root` → 用 root
  - 显式 `<用户名>` / `<uid>` → 用指定身份
- **服务级扩展**：默认身份 = 服务的身份配置（`user`/`uid`）
  - 扩展自身配置了 `run_as`/`run_as_uid` → 用扩展自身身份
  - 扩展未配置身份 → 继承服务的 `user`/`uid` 配置
- **服务自身**：`user`/`uid` 字段默认继承 supd 启动用户

**执行身份切换的实现**：

- 用 `cmd.SysProcAttr.Credential = &syscall.Credential{Uid: uid, Gid: gid}`
- User 模式：通过 `user.Lookup(name)` 获取 uid/gid，`group` 字段非空时通过 `user.LookupGroup` 覆盖主组 gid
- UID 模式：直接使用配置的 uid/gid/groups 数值
- **必须设置补充组**：User 模式通过 `user.LookupGroups` 获取目标用户的所有补充组；UID 模式使用配置的 `groups` 列表。`BuildCredential` 确保 groups 包含 gid 本身。禁止仅设置 Uid/Gid 而不设置补充组（如扩展脚本访问 docker.sock 需 docker 组，仅切 uid/gid 会失败）
- **必须 root 权限**才能切换到其他用户
- 非 root 启动 supd 时：
  - 服务：目标 uid ≠ 当前 uid → **拒绝启动**该服务，返回 `RUNTIME_USER_NOT_FOUND`（严格语义）
  - 扩展：目标 uid ≠ 当前 uid → **记录警告**，降级使用当前用户身份执行（宽松语义，区别于服务的严格拒绝）

**非 root 启动 supd 的限制**：

- 不能切换到其他用户（run_as / user / uid 字段限制）
- 不能绑定 1024 以下端口
- 不能修改文件权限（fix-attrs 类操作受限）
- 日志目录权限需提前设置好
- 配置了 run_as/user/uid 为其他用户的服务 → 拒绝启动该服务，提示具体服务名和配置位置

#### 2.2.14 扩展的展示规则

**扩展列表页（顶部导航"扩展"Tab）**：

- 展示所有全局扩展和服务级扩展
- 服务级扩展按服务分组展示（如"nginx 服务的扩展"、"mysql 服务的扩展"）
- 全局扩展单独一组展示

**服务详情页的"扩展"标签**：

- 仅展示该服务自己的服务级扩展
- 服务级扩展同时出现在扩展列表页和服务详情页（两处都展示，便于不同场景查看）

**展示规则**：

- 扩展 meta.yaml 中**有 actions 字段** → 展示卡片和按钮
- 扩展 meta.yaml 中**无 actions 字段** → 不展示按钮，但扩展仍在列表中显示（含状态、运行历史、触发时机等）
- 所有扩展（无论有无 actions）都在列表中展示，便于查看执行状态
- `enabled: false` 的扩展仍在列表中显示，但灰显并标注"已禁用"

**`enabled` 与 `actions` 的区别**：

- `enabled: false`：扩展完全不工作，不触发、不执行
- 无 `actions` 字段：扩展正常工作（仍可被 triggers 触发），只是前端无按钮

#### 2.2.15 全局扩展与服务级扩展的命名空间

- 全局扩展名必须在 `/etc/supd/extensions/` 下唯一
- 服务级扩展名必须在 `services/<svc>/extensions/` 下唯一
- 不同服务可以有同名扩展（如 nginx 和 mysql 都可以有名为 `backup` 的扩展）
- 全局扩展名可以与某服务的扩展名相同（在不同上下文，互不冲突）

#### 2.2.16 扩展运行日志上限

- 单次运行日志上限：10MB（硬编码，不配置化）
- 超过截断，尾部追加 `[日志超过 10MB，已截断]`
- 与任务历史同步清理（7 天）

#### 2.2.17 扩展系统约束

- **扩展间无依赖关系**：各自独立，不能声明依赖其他扩展
- **不实现 marketplace / 扩展商店 / 扩展搜索 / 扩展评分**
- **args 仅为字符串数组**，不引入 schema、类型校验、必填/可选标记
- **triggers 的 type 固定四种**：on_demand / on_schedule / service_lifecycle / supd_lifecycle，禁止新增类型
- **readiness.type 固定四种**：fd_notify / tcp_check / http_check / script，禁止新增
- **concurrency 固定四种**：replace / serialize / parallel / debounce:Ns（N 为 1-3600 的整数，单位秒），禁止其他格式
- **button_style 固定三种**：primary / default / danger，禁止新增
- **服务状态固定七种**：pending / starting / up / ready / stopping / down / failed
- **任务状态固定七种**：pending / running / success / failed / timeout / canceled / killed
- **禁止引入 namespace / cgroup / 其他隔离机制**（已明确不做沙箱）
- **禁止实现参数校验框架**（args 是字符串数组，扩展脚本自行解析）

### 2.3 配置文件规范

#### 2.3.1 config.yaml（全局主配置）

```yaml
settings:
  http_listen: ":8080"                 # HTTP 监听地址
  auth_mode: local_skip                # none | local_skip | always_token
  auth_token: "<token>"                # auth_mode != none 时必填
  local_networks:                      # auth_mode=local_skip 时生效

  log_level: info                      # debug | info | warn | error

  - env/00-base.yaml

  - extensions/
```

**字段说明**：

- 所有字段都有默认值，首次 `supd init` 生成完整 config.yaml
- 修改 `settings.http_listen` 和 `settings.auth_mode` 需重启 supd
- 其他字段保存后立即生效（按热重载规则）

#### 2.3.2 service.yaml（服务定义）

```yaml
name: <服务名>                         # 必填，与目录名一致，正则 ^[a-z][a-z0-9-]*$
version: <语义化版本>                  # 必填，如 1.0.0
description: <描述>                    # 可选
icon: <图标名>                         # 可选，前端图标库 key，默认 box
autostart: true                        # 可选，默认 true，supd 启动时是否自动拉起

command: [<命令数组>]                  # 必填
                                        #   省略 runtime：command[0] 是可执行文件，直接执行
                                        #   声明 runtime：command[0] 是传给 runtime 的第一个参数
runtime: <运行时别名>                  # 可选，省略时直接执行 command[0]；声明时用 runtime 解释 command
user: <用户名>                         # 可选，默认继承 supd 用户（User 模式，与 uid 互斥）
group: <组名>                          # 可选，默认同 user（User 模式下覆盖主组 gid，保留补充组）
uid: <数字>                            # 可选，直接指定 uid（UID 模式，与 user 互斥，不查 /etc/passwd）
gid: <数字>                            # 可选，直接指定 gid（0 或省略 = uid）
groups: [<gid 列表>]                  # 可选，补充组 gid 列表（UID 模式）
workdir: <绝对路径>                    # 可选，默认 /etc/supd/services/<name>/；禁止相对路径（校验拒绝非 / 开头）

depends_on: [<服务名列表>]            # 可选，默认空
tags: [<标签列表>]                    # 可选

readiness:                             # 可选，无则进程启动即 up
  type: <fd_notify | tcp_check | http_check | script>
  fd: 3                                # type=fd_notify 时
  port: 80                             # type=tcp_check 时
  url: <URL>                           # type=http_check 时
  expected_status: 200                 # type=http_check 时，默认 200
  check: [<命令数组>]                  # type=script 时，继承服务的环境变量
  interval_seconds: 1                  # 默认 1
  timeout_seconds: 5                   # 默认 5

restart:                               # 可选，覆盖全局默认
  policy: <always | on-failure | never>
  backoff_ms: <int>
  max_backoff_ms: <int>
  multiplier: <int>
  max_retries: <int>
  reset_after_seconds: <int>

stop:                                  # 可选
  grace_seconds: 10                    # SIGTERM 后等几秒再 SIGKILL（默认 10）
  timeout_seconds: 60                  # 整个停止流程上限
                                        #   注：supd 整体关机时受全局 shutdown_grace_seconds 限制
                                        #   单服务 grace 不得超过剩余的全局 grace

logging:                               # 可选
  enabled: true                        # 默认 true
  max_size_mb: 10                      # 默认 10
  max_files: 5                         # 默认 5

signals:                               # 可选，自定义信号按钮
                                        #   允许：HUP / INT / QUIT / USR1 / USR2 / PIPE / ALRM / CHLD 共 8 种
                                        #   禁止：TERM / KILL / STOP / CONT / SEGV / ABRT / BUS / FPE / ILL（系统保留）
  reload: HUP
  rotate_logs: USR1
  graceful_quit: QUIT

package:                               # 可选，打包配置
  include: [<路径列表>]               # 默认含 service.yaml, env.yaml, extensions/
  exclude: [<路径或通配符>]           # 默认含 data/, logs/, *.log, .cache/
  default: include                     # include | exclude
```

**autostart 字段说明**：

- `autostart: true`（默认）：supd 启动时自动按依赖图拉起该服务
- `autostart: false`：supd 启动时不拉起，需用户手动启动
- autostart 仅控制 supd 启动时的初始行为，不影响运行时的 restart.policy。用户手动启动后，服务崩溃仍按 restart.policy 自动重启
- 用户手动停止后服务状态变为 down，supd 重启后会按 autostart 配置再次拉起

#### 2.3.3 env.yaml（环境变量文件）

```yaml
env:
  <VAR_NAME>:
    value: <字符串值>
    enabled: true                      # 可选，默认 true
    hint: <说明文字>                   # 可选
```

适用于全局 env、服务 env、扩展 env。

#### 2.3.4 数据模型约束

- **禁止引入任何数据库**：SQLite / bolt / badger / bbolt 等均禁止
- **任务历史仅存内存**：不持久化（supd 重启后清空）
- **文件历史用简单文件存储**：`/var/lib/supd/history/<path-hash>/v<N>.yaml`，不引入索引文件、不引入 manifest
- **禁止 JSON Schema 动态生成**：配置校验用代码内置 struct tag，不读取外部 schema 文件
- **配置文件不支持继承 / include / 引用**：每个配置文件独立
- **禁止配置版本自动迁移逻辑**：不做旧版本到新版本的自动转换

### 2.4 文件发现与热重载

#### 2.4.1 文件发现规则

- `services/*/service.yaml` 存在 → 注册为服务
- `extensions/*/meta.yaml` 存在 → 注册为全局扩展
- `services/<svc>/extensions/*/meta.yaml` 存在 → 注册为服务级扩展
- `runtimes/*` 是可执行文件 → 注册为可用运行时
- `env/*.yaml` 存在 → 加载为全局环境变量

#### 2.4.2 fsnotify 监听

- 监听 `/etc/supd/` 下所有子目录
- **防抖**：500ms 内的连续事件合并为一次重载
- **按文件路径去重**：同一文件的多次变更只触发一次重载
- **删除事件**：服务/扩展目录被删 → 卸载该服务/扩展，不报错
- **原子写支持**：监听 rename 事件而非 create+write，避免读到半写文件

#### 2.4.3 配置热重载规则

| 配置项 | 修改后行为 |
|---|---|
| service.yaml: command / user / group / workdir | 需重启服务 |
| service.yaml: depends_on | 下次启动生效 |
| service.yaml: readiness | 下次启动生效 |
| service.yaml: restart | 立即生效 |
| service.yaml: logging | 立即生效 |
| service.yaml: tags | 立即生效 |
| service.yaml: stop | 立即生效 |
| service.yaml: signals | 立即生效 |
| service.yaml: package | 立即生效（影响下次导出） |
| env.yaml（服务） | 需重启服务 |
| meta.yaml: triggers | 立即生效 |
| meta.yaml: actions | 立即生效 |
| meta.yaml: timeout | 下次运行生效 |
| meta.yaml: run_as | 下次运行生效 |
| meta.yaml: concurrency | 立即生效 |
| env.yaml（扩展） | 下次运行生效 |
| config.yaml: settings.http_listen | 需重启 supd |
| config.yaml: settings.auth_* | 需重启 supd |
| config.yaml: settings.local_networks | 需重启 supd |
| config.yaml: settings.log_* | 立即生效 |
| config.yaml: settings.shutdown_grace_seconds | 立即生效 |
| config.yaml: settings.extension_* | 立即生效 |
| config.yaml: settings.max_upload_size_mb | 立即生效 |
| config.yaml: settings.run_history_retention_seconds | 立即生效 |
| config.yaml: settings.file_history_versions | 立即生效 |
| config.yaml: defaults.restart | 立即生效 |
| env/*.yaml（全局） | 不影响已运行服务；新启动的服务用新 env |

**热重载与运行中服务的冲突处理**：

- 设计原则：热重载不中断正在运行的服务和脚本，让它们继续正常运行
- "需重启服务"类型的变更（command / user / group / workdir）标记为"待生效"，不自动重启服务，下次服务重启时应用
- 服务处于 `starting` 时，变更同样标记为"待生效"，等服务进入稳定状态（up/ready/failed/down）后再应用
- WebUI 在服务详情页对"待生效"的变更显示提示"配置已更新，重启服务后生效"
- "立即生效"类型的变更（restart / logging / tags / stop / signals / package）直接应用，不影响运行中的进程

### 2.5 运行时管理

#### 2.5.1 三层运行时来源

运行时（runtime）是执行服务命令或扩展脚本的解释器/二进制。supd 通过三层来源解析运行时别名到绝对路径：

| 优先级 | 来源 | 说明 |
|---|---|---|
| 1（最高） | config.yaml 的 `runtimes` 配置 | 用户显式声明的别名 → 绝对路径映射 |
| 2 | `/etc/supd/runtimes/` 目录扫描 | 文件名即别名，绝对路径即路径 |
| 3（最低） | 内置默认 | 开箱即用的常见运行时 |

同名别名时高优先级覆盖低优先级。

#### 2.5.2 内置默认运行时

supd 二进制内置以下运行时，开箱即用：

| 别名 | 路径 |
|---|---|
| `bash` | `/bin/bash` |
| `sh` | `/bin/sh` |
| `python3` | `python3`（PATH 查找） |
| `node` | `node`（PATH 查找） |

#### 2.5.3 config.yaml 配置覆盖

用户在 config.yaml 中可声明/覆盖运行时别名：

```yaml
runtimes:
  tjs: /opt/tkixijs/tkixijs              # 新增别名
  bun: /etc/supd/runtimes/bun      # 显式声明 runtimes/ 下的
  deno: /etc/supd/runtimes/deno
  python3: /usr/bin/python3.11           # 覆盖内置 python3 的路径
  node: /usr/local/bin/node              # 覆盖内置 node 的路径
  custom-js: /home/user/my-runtime.sh    # 任意可执行文件
```

#### 2.5.4 runtimes/ 目录扫描

- 扫描 `/etc/supd/runtimes/` 下的可执行文件
- 文件名即别名，绝对路径即路径
- 与 config.yaml 的 runtimes 配置合并（config.yaml 优先级更高）
- 不在 config.yaml 中显式声明的 runtimes/ 文件，自动按"文件名 → 绝对路径"注册
- 单文件，需要换版本直接覆盖
- 不做版本管理（家庭场景不需要）

#### 2.5.5 runtime 字段的使用

**meta.yaml 和 service.yaml 都支持 `runtime` 字段**，引用别名：

```yaml
runtime: bash        # 默认值，可省略
runtime: tjs         # 引用别名
runtime: bun
runtime: python3
```

**调用规则**：

- **省略 runtime**（默认）：直接执行 `command[0] command[1] ...`，适合二进制程序直接启动
- **声明 runtime**：执行 `<runtime 路径> command[0] command[1] ...`，适合脚本启动

**`runtime` 引用的别名不存在** → 启动/运行失败，返回 `RUNTIME_NOT_FOUND` 错误，提示具体别名和配置位置

#### 2.5.6 完整调用链示例

**服务调用示例**：

| runtime | command | 实际执行命令 |
|---|---|---|
| （省略） | `["nginx", "-g", "daemon off;"]` | `nginx -g "daemon off;"` |
| （省略） | `["/usr/local/bin/myapp", "--port", "8080"]` | `/usr/local/bin/myapp --port 8080` |
| `bash` | `["run.sh", "arg1"]` | `/bin/bash run.sh arg1` |
| `bun` | `["app.ts", "--port", "3000"]` | `/etc/supd/runtimes/bun app.ts --port 3000` |
| `node` | `["app.js"]` | `/usr/local/bin/node app.js` |
| `tjs` | `["run.js", "--verbose"]` | `/opt/tkixijs/tkixijs run.js --verbose` |

**扩展调用示例**（扩展的 entry 等同于服务的 command[0]）：

| runtime | entry | args | 实际执行命令 |
|---|---|---|---|
| `bash`（默认） | `run.sh` | `["backup"]` | `/bin/bash run.sh backup` |
| `bun` | `run.ts` | `["backup"]` | `/etc/supd/runtimes/bun run.ts backup` |
| `tjs` | `run.js` | `["--verbose"]` | `/opt/tkixijs/tkixijs run.js --verbose` |
| `python3` | `run.py` | `["backup"]` | `/usr/bin/python3.11 run.py backup` |

工作目录设为扩展目录或服务 workdir，所以 entry 用相对路径即可。

#### 2.5.7 运行时可用性校验

- 服务/扩展启动前检查 runtime 别名是否能解析到路径
- 检查路径文件是否存在且可执行
- 不存在或不可执行 → 启动失败，返回 `RUNTIME_NOT_FOUND` 或 `RUNTIME_NOT_EXECUTABLE` 错误
- **不做版本检测**

#### 2.5.8 运行时路径不注入 PATH

- 选中的 runtime 路径仅用于执行 entry
- **不**把 runtimes/ 目录加入子进程的 PATH
- 如果扩展脚本依赖其他 runtimes/ 下的工具，需在脚本里显式引用绝对路径
- 子进程的 PATH 来自全局 env 的 PATH 变量

#### 2.5.9 前端运行时选择

服务/扩展创建表单提供运行时下拉：

```
运行时  [bash                              ▼]
        ┌──────────────────────────────────────┐
        │ bash             (内置, /bin/bash)   │
        │ sh               (内置, /bin/sh)     │
        │ python3          (配置, /usr/bin/...) │
        │ node             (配置, /usr/local/.) │
        │ bun              (配置, /etc/superv.) │
        │ deno             (扫描, /etc/superv.) │
        │ tjs              (配置, /opt/tkixijs) │
        │ ───────────────────────────          │
        │ (省略, 直接执行二进制)                │
        └──────────────────────────────────────┘
```

- 默认选中 bash
- 不可用的运行时灰显但可选择（用户能看到配置问题）
- 选择"省略"时表示直接执行 command[0]，不用 runtime
- 选择后 entry 字段提示对应文件类型

#### 2.5.10 runtimes/ 目录的 fsnotify 监听

- runtimes/ 目录被 fsnotify 监听
- 新增/删除/重命名二进制 → 自动更新运行时列表
- 前端通过轮询 `/api/runtimes` 感知变化（10 秒一次，进入相关页面时）

#### 2.5.11 运行时管理 API

- `GET /api/runtimes` — 列出所有运行时（含 source 和 available 字段）
- `POST /api/runtimes/upload` — 上传运行时二进制到 runtimes/ 目录
- `DELETE /api/runtimes/<name>` — 删除 runtimes/ 下的运行时
- `PUT /api/settings/runtimes` — 更新 config.yaml 的 runtimes 配置

### 2.6 通信与 API

#### 2.6.1 通信原则

- **不使用 SSE / WebSocket**——统一 HTTP
- **实时数据**用长轮询（30 秒挂起，有数据立即返回）
- **普通数据**用短轮询（1-5 秒）
- 前端用全局轮询调度器，避免重复请求

#### 2.6.2 API 风格

- RESTful 资源导向，URL 表达资源，HTTP 方法表达操作
- 所有响应 JSON 格式
- **API 不做版本化**：家庭场景，破坏性变更随 supd 版本升级，不保留旧 API
- **列表端点不分页**（services、extensions、runtimes、cron 规模小），事件流和任务列表靠 since 游标增量拉取
- **列表端点不提供过滤/排序参数**：前端客户端过滤/排序
- **文件上传统一用 multipart/form-data**：禁止用 base64 in JSON
- **错误响应不添加 trace_id / request_id / span_id** 等追踪字段
- **长轮询不引入 ETag、If-Modified-Since、压缩**等机制
- **禁止引入 GraphQL、gRPC、WebSocket**
- **校验错误响应支持多字段**：

```json
{
  "error": {
    "code": "<ERROR_CODE>",
    "message": "<人类可读信息>",
    "details": {
      "fields": [
        {"field": "command", "message": "必须是数组"},
        {"field": "user", "message": "用户不存在"}
      ]
    }
  }
}
```

单字段错误时 fields 含 1 个元素。

**长轮询并发上限**：

- 单客户端最多 5 个并发长轮询
- 全局最多 50 个
- 超限返回 503 + SERVICE_BUSY 错误码

**通用错误响应格式**（非校验错误）：

```json
{
  "error": {
    "code": "<ERROR_CODE>",
    "message": "<人类可读信息>",
    "details": {<可选上下文>}
  }
}
```

#### 2.6.3 HTTP 状态码使用

| 状态码 | 场景 |
|---|---|
| 200 | 请求成功 |
| 201 | 创建成功 |
| 400 | 请求参数错误 |
| 401 | 未认证或 token 无效 |
| 403 | 禁止访问（权限不足或路径不在白名单） |
| 404 | 资源不存在 |
| 409 | 资源冲突 |
| 413 | 上传文件过大 |
| 422 | 业务逻辑错误（如循环依赖、配置校验失败） |
| 500 | 服务端内部错误 |
| 503 | 服务繁忙（长轮询并发超限） |

#### 2.6.4 长轮询端点规范

长轮询端点的请求参数：

```
GET /api/...?since=<ISO8601时间戳>&wait=30&limit=500
```

- `since`：ISO8601 带时区时间戳，表示从此时间之后拉取新数据。supd 重启后单调时钟重置，前端通过 `GET /api/system/status` 检测启动时间变化后丢弃旧 since 值，从当前时间重新拉取
- `wait`：整数秒（如 `wait=30`），不用 Go Duration 字符串；默认 30，最大 60
- `limit`：单次返回最大条数，默认 500

响应：

```json
{
  "data": [...],
  "next_since": "<新的时间戳>",
  "has_more": false
}
```

- `wait` 默认 30 秒，最大 60 秒
- 30 秒内无新数据 → 返回空 data
- 有数据立即返回
- `has_more=true` 时客户端立即发起下一次请求

- 首次 `supd init` 随机生成 token，写入 config.yaml

- 前端登录后存 localStorage
- 设置页可重新生成 token，所有客户端需重新登录

### 2.8 优雅退出

#### 2.8.1 supd 退出流程

收到 SIGTERM / SIGINT 时：

1. 触发 `supd_lifecycle:pre_shutdown` 扩展
2. 标记"停止接受新请求"
3. 按依赖反序停止所有运行中的服务（每个服务执行完整停止流程，含 pre_stop 扩展，受各自的 stop.timeout_seconds 约束）
   - **同层服务并行停止**（不串行），同层全部停止后才停止下一层
   - 同层并行停止的总时长 = max(各服务 stop.timeout_seconds)
   - 每个服务：pre_stop 扩展 → SIGTERM → 等 `stop.grace_seconds` → SIGKILL
4. 等待所有运行中的扩展任务结束（最多等 30 秒）
5. 关闭 HTTP 服务器
6. 退出

**全局 grace 期**：`shutdown_grace_seconds`（默认 30 秒）控制整个关机流程上限。关机流程总时长 = 服务停止总时长 + 扩展任务等待 30 秒，受此上限约束。超过上限时强制 SIGKILL 所有剩余进程，supd 立即退出。

**启动中收到信号的处理**：

- 启动中（服务正在拉起）收到 SIGTERM/SIGINT 时，立即停止拉起剩余服务
- 已启动的服务按正常关机流程停止

**关机超时强制退出**：

- 关机超时（shutdown_grace_seconds 到期）强制退出前，supd_lifecycle:pre_shutdown 已执行过的扩展无法取消
- 记录警告日志并强制 SIGKILL 所有剩余进程

#### 2.8.2 启动时状态恢复策略

**零状态启动**：

- 无论 supd 上次如何退出，重启后所有服务回到 `autostart` 字段定义的初始状态
- `autostart: true` 的服务按依赖图拉起
- `autostart: false` 的服务保持 `down` 状态
- 用户手动停止的状态**不持久化**，supd 重启后按 autostart 重置
- 运行中的扩展任务**丢弃，不恢复**
- **禁止**引入任何状态持久化文件记录"手动停止"等运行时状态

**用户预期提示**：

supd 重启后所有服务按 autostart 重置，手动停止状态不保留。WebUI 在 Dashboard 顶部显示一次性提示横幅"supd 已重启，所有服务按 autostart 配置重置"，用户关闭后本次会话不再显示，下次 supd 重启后再次出现。

#### 2.8.3 supd 启动流程

supd 启动时按以下顺序初始化：

1. 读取 config.yaml（解析失败则拒绝启动）
2. 初始化日志系统（log/slog + 文件输出）
3. 加载运行时：合并内置默认 + config.yaml 的 runtimes 配置 + 扫描 runtimes/ 目录
4. 扫描配置：services/ + extensions/ + env/ + 服务级 extensions/
5. 构建依赖图 + 循环检测（含自引用检测）
6. 启动 fsnotify 监听
7. 启动 cron 调度器（加载 on_schedule 触发器）
8. 启动 HTTP 服务器
9. 触发 `supd_lifecycle:pre_start` 扩展
10. 按 autostart + 依赖图启动服务（同层并行，跨层等待 ready）
11. 所有 autostart=true 的服务进入终态（ready 或 failed）后触发 `supd_lifecycle:post_ready` 扩展。autostart=false 的服务不参与此判定

**启动失败处理**：

- config.yaml 解析失败 → 拒绝启动，提示错误位置
- 单个 service.yaml / meta.yaml / env.yaml 解析失败 → 标记该项为"配置错误"状态，不影响其他服务和 supd 启动
- fsnotify 初始化失败 → 拒绝启动（见 2.14.6）

**事件 JSON 统一 Schema**：

每条事件一行 JSON，字段固定：

```json
{
  "ts": "2026-07-07T15:30:00+08:00",
  "type": "service_died",
  "source": "service:nginx",
  "severity": "warn",
  "message": "nginx 进程退出",
  "details": {
    "pid": 12345,
    "exit_code": 1,
    "signal": 0
  }
}
```

**基础字段**：

- `ts`：ISO8601 带时区
- `type`：事件类型（见枚举）
- `source`：事件来源（如 `service:nginx` / `extension:backup` / `system`）
- `severity`：`info` / `warn` / `error`
- `message`：人类可读摘要
- `details`：类型相关的上下文

**details 字段约定**（按事件类型）：

- `service_state`：`from` / `to` / `pid`
- `service_died`：`pid` / `exit_code` / `signal`
- `service_ready`：`pid` / `check_type` / `check_duration_ms`
- `service_failed`：`pid` / `reason`
- `service_exited`：`pid` / `exit_code` / `duration`
- `extension_started`：`run_id` / `action` / `trigger`
- `extension_completed`：`run_id` / `duration_ms` / `status`
- `extension_failed`：`run_id` / `reason` / `exit_code`
- `extension_canceled`：`run_id` / `reason`
- `extension_timeout`：`run_id` / `timeout_seconds`
- `cron_triggered`：`extension` / `action` / `schedule`
- `config_reloaded`：`path` / `change_type`
- `config_reload_failed`：`path` / `error` / `line`
- `system_resource_warning`：`resource_type`（disk / fd / memory） / `message`

**禁止**：

- 禁止在事件 JSON 中添加 `trace_id` / `request_id` 等追踪字段（家庭场景不需要分布式追踪）
- 禁止在 details 中嵌套超过 2 层（如 `details.service.name` 应拍平为 `details.service_name`）

**事件流的存储**：

- 内存环形缓冲：200 条，超出丢最旧
- 落盘文件：`/var/log/supd/events.log`，每条事件一行 JSON
- 落盘文件轮转策略：与 supd.log 一致（max_size_mb + max_files）

#### 2.9.8 服务列表的排序与筛选

- **默认排序**：按名称字母序
- **可切换排序**：按状态、按运行时长、按重启次数
- **筛选**：按 tag、按状态
- **批量操作**：可多选后批量启停；按 tag 批量启停也支持

#### 2.9.9 文件页面

- 独立入口的全屏 Monaco 编辑器
- 文件树作为可滑出抽屉（默认隐藏，点 📁 按钮唤出）
- 多标签支持（最多 8 个）
- 路径白名单限制

#### 2.9.10 设置页面

单页滚动 + 分段锚点导航，覆盖 config.yaml 全部字段：

- **基础设置**：HTTP 监听、认证模式、Token、内网网段
- **日志**：单文件上限、保留文件数、日志级别
- **关机**：grace 期
- **扩展执行**：默认超时、全局硬上限、任务历史保留、文件历史版本数
- **文件上传**：最大上传大小
- **重启策略默认值**：策略、退避参数
- **全局环境变量**：跳转到独立编辑器
- **关于**：版本、构建信息、工作目录、检查更新、导出诊断信息

保存行为：所有改动需点"保存"按钮才生效；不支持自动保存；某些字段改动需重启 supd，保存时提示。

#### 2.9.11 WebUI 实现约束

**zustand 状态管理**：

- 建议按域分少量 store（如 serviceStore / extensionStore / eventStore / settingsStore），数量克制
- 禁止过度拆分（如每页面一个 store）
- 禁止引入 redux / redux-toolkit / mobx / jotai

**tanstack-query**：

- 使用默认配置，不引入乐观更新、自定义缓存策略
- 使用 `staleTime: 0` 的默认值，每次都重新拉取
- 不引入复杂的 query key 设计

**shadcn/ui**：

- 不深度定制（仅使用库提供的组件，不重写内部逻辑）
- 不自定义主题、动画、过渡

**react-router**：

- 使用最基础的 Routes/Route 配置
- 禁止使用 loader/action、嵌套路由、defer

**动画**：

- 仅用 TailwindCSS 的 transition-* 类
- 禁止用 CSS @keyframes 实现复杂动画
- 已禁止 framer-motion / react-spring / GSAP

**Error Boundary**：

- 每个路由级组件一个 Error Boundary（如 ServiceDetail、ExtensionDetail、Settings 等）
- 不在子组件级嵌套
- 崩溃时显示"页面出错，请刷新"+ 错误摘要 + "刷新页面"按钮

**轮询调度器**：

- 单例（zustand store 内的一个调度器模块）
- 所有轮询注册到调度器，调度器统一管理频率和取消
- 页面切换时，旧页面的轮询自动取消
- 轮询频率不可配置，硬编码（长轮询 30s、短轮询 2s、资源采集 10s）

#### 2.9.12 WebUI 设计系统

**设计原则**：基于 shadcn/ui 原生风格，不深度定制主题、不自定义动画、不引入额外 CSS 文件。所有视觉元素尽量复用 shadcn/ui 默认值，仅在状态徽标配色、图标映射等必要处补充约定。目标是"美观大方但克制"，避免过度设计造成风格冲突。

### 2.10 可视化表单

#### 2.10.1 设计原则

- 单页滚动展示所有字段
- 左侧分段锚点导航（点击平滑滚动，滚动时高亮当前段）
- 所有字段在同一页面可见，用户可一次填完
- 有规律的字段预填默认值
- 字段联动（如选 TCP 才显示端口号）
- 实时校验（失焦时校验，错误标红）
- 创建和修改共用同一表单，区别在标题和按钮文字

#### 2.10.2 服务创建表单字段

分段：基本信息 / 进程配置 / 高级配置

**基本信息**：

| 字段 | 类型 | 默认值 | 必填 | 校验 |
|---|---|---|---|---|
| 服务名 | 文本框 | （空） | 是 | `^[a-z][a-z0-9-]*$`，唯一 |
| 版本 | 文本框 | `1.0.0` | 否 | `\d+\.\d+\.\d+` |
| 描述 | 文本框 | （空） | 否 | - |
| 标签 | 标签输入 | （空） | 否 | - |

**进程配置**：

| 字段 | 类型 | 默认值 | 必填 |
|---|---|---|---|
| 启动命令 | 文本框 | （空） | 是 |
| 工作目录 | 文本框 | `/etc/supd/services/<name>` | 否 |
| 运行用户 | 下拉+自定义 | 当前 supd 用户 | 否 |
| 用户组 | 下拉+自定义 | 同运行用户 | 否 |

**高级配置**：

- 依赖服务：复选框列表（已存在的服务）
- 就绪检查：类型单选 + 字段联动
- 重启策略：策略单选 + 退避参数
- 日志：启用开关 + 上限 + 保留数
- 信号：自定义信号按钮
- 停止：grace + 超时

#### 2.10.3 扩展创建表单字段

分段：基本信息 / 执行配置 / 触发时机 / Actions

**基本信息**：扩展名、版本、描述、所属范围（全局/服务级）、服务（服务级时显示）、图标、分类

**执行配置**：解释器、入口文件、超时、执行身份、并发策略（debounce 时显示防抖时长）

**触发时机**（可多选）：
- 手动触发（on_demand）
- 定时触发（on_schedule）：Cron 表达式 + 实时下次执行预览 + 失败重试配置
- 服务生命周期（service_lifecycle）：阶段复选 + 服务复选 + action
- supd 生命周期（supd_lifecycle）：阶段复选 + action

**Actions**：动态列表，每项含 ID、标签、图标、样式、CLI args、启用开关

#### 2.10.4 创建后行为

1. 后端生成 service.yaml / meta.yaml
2. 后端生成空 env.yaml
3. 后端生成模板 run.sh（扩展才有，含 action dispatch 框架和 ::progress:: / ::result:: 示例）
4. 后端创建空 extensions/ 目录（服务才有）
5. fsnotify 触发热重载
6. 前端跳转到编辑器，打开刚创建的文件
7. 提示"已创建，请在编辑器补充脚本"

### 2.11 代码编辑器

#### 2.11.1 技术实现

- 基于 Monaco Editor + @monaco-editor/react + monaco-yaml
- YAML 文件通过 JSON Schema 提供字段校验、自动补全、悬停提示
- 自定义补全：`depends_on` 字段补全已有服务名、脚本中 `$` 补全环境变量、`::` 补全协议指令

#### 2.11.2 编辑器布局

- 抽屉式：文件树默认隐藏，点 📁 按钮唤出
- 文件树从左侧滑出，覆盖在编辑器上
- 点击文件 → 抽屉收起 → 编辑器加载新文件
- 图钉按钮（📌）让抽屉常驻
- 顶部显示当前文件路径（相对 + 绝对）
- 多标签支持（最多 8 个），未保存的有 ● 提示
- 标签状态 localStorage 持久化

#### 2.11.3 编辑器嵌入位置

- 服务详情页 → 配置标签（仅该服务文件）
- 扩展详情页 → 代码标签（仅该扩展文件）
- 文件页面（全屏，所有 /etc/supd/ 文件）
- 设置页 → 全局配置（config.yaml + env/）

#### 2.11.4 文件保存与热重载

1. 用户编辑文件
2. Ctrl+S 保存（或点保存按钮）
3. 前端 PUT /api/files，body 含 path 和 content
4. 后端写入磁盘（原子写：临时文件 + rename）
5. fsnotify 触发热重载
6. 后端返回 `{applied: true, requires_restart: bool, warnings: [], errors: []}`
7. 前端在编辑器右下角显示"已保存并应用"

#### 2.11.5 文件历史版本

- **不自动保存**，仅在用户主动点"创建快照"时保存
- 快照存储位置：`/var/lib/supd/history/<path-hash>/v<N>.yaml`
- 保留最近 50 版（`file_history_versions` 可配）
- 可回滚到任意历史版本
- 快照内容含创建时间、可选备注

#### 2.11.6 编辑器不自动保存策略

- 编辑器内容 vs 磁盘内容的差异显示（顶栏 ● 提示）
- 切换标签时未保存的内容保留在内存
- 关闭浏览器时未保存的内容丢失（不做草稿持久化）
- 保存按钮始终可点（即使无修改也允许覆盖保存）

#### 2.11.7 配置校验时机

- **文件保存时校验**：POST /api/files/validate 接口，返回 errors 和 warnings
- **supd 启动时校验**：所有配置文件，校验失败的不阻塞其他配置加载（只标记该项错误）
- **校验失败的配置**：对应服务/扩展标记为"配置错误"状态，不启动
- **校验失败不阻塞 supd 启动**：其他正常配置照常工作

#### 2.11.8 错误响应含位置信息

YAML 解析错误、字段校验错误必须含位置信息：

```json
{
  "error": {
    "code": "SERVICE_CONFIG_INVALID",
    "message": "service.yaml 第 5 行: command 字段必须是数组",
    "details": {
      "file": "services/nginx/service.yaml",
      "line": 5,
      "field": "command"
    }
  }
}
```

- `file`：相对工作目录的文件路径
- `line`：YAML 行号（从 1 开始）
- `field`：出错字段名（可选）

#### 2.11.9 不做断点调试

- 评估结论：无成熟方案，工程量大
- 替代方案：扩展详情页提供"试运行"功能
  - 选择 action
  - 临时覆盖环境变量（不保存到 env.yaml）
  - 自定义超时
  - 实时看日志和进度
  - 不记入正式运行历史

### 2.12 导入导出

#### 2.12.1 打包格式

.tar.gz 文件，命名 `<name>-v<version>.tar.gz`

#### 2.12.2 服务打包

- 服务目录所有文件按 `package.include` / `package.exclude` / `package.default` 规则过滤
- 强制包含：`service.yaml`、`env.yaml`、`extensions/`
- 强制排除：`data/`、`logs/`、`*.log`、`.cache/`、`*.tmp`、`*.bak`
- `default: include`（默认）：扫描全部，去掉 exclude 中的
- `default: exclude`：仅打包 include 中的

#### 2.12.3 扩展打包

- 全量打包整个扩展目录
- 强制包含：`meta.yaml`、`env.yaml`、entry 声明的文件
- 强制排除：`*.log`、`.cache/`、`__pycache__/`、`node_modules/`、`*.tmp`、`*.bak`
- **meta.yaml 不含 package 配置字段**（简化）

#### 2.12.4 版本信息

- 不使用独立 manifest.json
- 版本号直接读自 service.yaml 的 `version` 字段和各扩展 meta.yaml 的 `version` 字段

#### 2.12.5 导入流程

1. 用户上传 .tar.gz
2. 后端解压到临时目录
3. 读 service.yaml 和各 meta.yaml 的 version
4. 与本地已有版本对比，生成对比信息
5. 前端弹窗展示对比：

```
服务: nginx        本地 v1.1.0  →  导入 v1.2.0   [升级]
扩展:
  reload-on-cert   本地 v1.0.0  →  导入 v1.0.0  [相同]
  cleanup-tmp      本地 v0.2.0  →  导入 v0.3.0  [升级]
  new-feature      (新增)
  old-hook         本地 v0.1.0  →  (已删除)
```

6. 用户确认后：
   - 备份当前目录到 `<name>.bak.<timestamp>/`
   - 用导入文件覆盖
   - 触发 fsnotify 热重载
   - 提示"已导入，建议重启服务使配置生效"

#### 2.12.6 上传大小限制

- 全局配置 `max_upload_size_mb`（默认 100MB）
- 适用于：服务导入、扩展导入、运行时上传、单文件上传
- 超限返回 `FILE_TOO_LARGE` 错误

### 2.14 安全性

#### 2.14.1 路径校验

- 所有文件操作必须 `filepath.Abs` + 检查 `..` + 白名单前缀
- 文件 API 严格限制可访问路径（见 2.6.6）

#### 2.14.2 YAML 解析

- 使用 `go.yaml.in/yaml/v4`
- 限制解析深度（防止 YAML billion laughs 攻击）

#### 2.14.3 命令注入

- service.yaml 的 command 用数组而非字符串，避免 shell 解析
- 扩展的 args 用数组传递，不经 shell

#### 2.14.4 扩展执行

- 扩展是受信任的，等于 root 权限
- 不做沙箱
- 通过 run_as 切换用户实现权限隔离

#### 2.14.5 上传文件

- 限制大小（`max_upload_size_mb`）
- 解压时检查路径不含 `..`（防 zip slip）

#### 2.14.6 异常处理

**fsnotify 失败**：

- fsnotify 初始化失败时 supd 拒绝启动（返回错误并退出）
- fsnotify 运行中失败（如 fd 耗尽）时记录错误日志，禁用热重载，其他功能正常
- **不实现重试逻辑**

**磁盘满**：

- 日志写入失败：保留内存缓冲（最近 1000 行），轮转跳过，不报错给调用方
- 事件落盘失败：仅保留内存环形缓冲
- 配置保存失败：返回 500 + INTERNAL_ERROR
- 导入失败：返回 500 + INTERNAL_ERROR，清理临时文件

**user.Lookup 失败**：

- run_as 指定的用户不存在 → 返回 422 + RUNTIME_USER_NOT_FOUND
- 错误响应含具体用户名和配置位置

**YAML 解析防护**：

- 解析深度限制 100 层（超过拒绝解析并返回错误）
- 别名数限制 50 个（超过拒绝解析）
- 这些限制在配置加载时检查，不在每次读取时检查

**资源耗尽场景（通过事件 + 全局扩展处理）**：

- 磁盘满、fd 耗尽、内存压力等场景，supd 触发对应事件（如 `system_resource_warning`）
- 不在程序内部实现重试或自动处理
- 用户通过全局扩展（订阅对应事件）实现相关处理逻辑（如清理临时文件、发通知等）
- 这符合"扩展即一等公民"原则，保持 supd 核心简单

### 4.3 "执行"按钮的统一语义

**按钮显示规则**（与 2.2.14 一致）：

- 扩展 meta.yaml 中**有 actions 字段**：每个 action 对应一个按钮，按钮文字 = action 的 label，按钮样式 = action 的 button_style
- 扩展 meta.yaml 中**无 actions 字段**：不展示执行按钮，扩展仍可被 triggers 触发运行（lifecycle / schedule）
- 单个 action 时直接显示该 action 的按钮，不显示下拉
- 多个 action 时按 button_style 优先级排序（primary 在前，default 居中，danger 在后），平铺展示，不显示下拉

**点击行为**：

- 点击 action 按钮 → 触发对应 action（执行 `<runtime> entry args...`）
- 始终可点（不阻塞）
- 同 action 有任务运行时按钮加细边框提示
- 点击运行中的 action → 按并发策略处理（默认 replace）

执行后展开抽屉或浮窗，展示：
- 实时日志流
- 进度条（如果脚本输出 ::progress::）
- 最终结果（::result::）
- 退出码
- 耗时
- 取消按钮（运行中时）

### 4.4 任务条与浮窗

**任务条**：扩展卡片下方，展示该扩展的运行中和最近完成任务

**全局浮窗**：底部常驻，所有页面可见，展示所有扩展的运行任务

**数据源**：全局轮询 `/api/extensions/runs?include_recent=true`，2 秒一次

**按钮状态**：

- 始终可点（不阻塞）
- 同 action 有任务运行时按钮加细边框提示
- 点击运行中 action → 按并发策略处理（默认 replace）

## 第五部分：实施约束

### 5.1 编码规范

- Go 代码遵循 `gofmt` + `golangci-lint`
- TypeScript 代码遵循 ESLint + Prettier
- 所有公开类型和函数有文档注释
- 错误处理用 `errors.Is` / `errors.As`，禁止 `==` 比较
- 上下文用 `context.Context` 传递，禁止全局变量

### 5.2 测试约束

**验收标准**：

- 状态机所有转移路径至少 1 个测试（含正常路径和异常路径）
- 依赖图循环检测、拓扑排序、同层并行启动各至少 1 个测试
- env 合并顺序、同名覆盖、enabled=false 不注入各至少 1 个测试
- stdout 协议解析的正常路径、容错路径（百分比非数字、状态非枚举、超长行截断、未识别前缀）各至少 1 个测试
- 每个 API 端点至少 1 个集成测试

**禁止**：

- 禁止引入 testify / gomock / mockery 等测试框架，统一用 Go 标准库 testing 包
- 禁止为了提高覆盖率写无意义的 getter / setter 测试

### 5.3 文件命名规范

- Go 文件：小写 + 下划线（如 `service_manager.go`）
- 前端文件：PascalCase（如 `ServiceDetail.tsx`）
- YAML 文件：小写（如 `service.yaml`）
- 日志归档：`@<ISO8601>.s`（如 `@2026-07-06T15-30-00.s`）

### 5.4 错误码与 HTTP 状态码完整映射

| 错误码 | HTTP | 触发场景 |
|---|---|---|
| `AUTH_REQUIRED` | 401 | 未提供 token 或 token 无效 |
| `AUTH_INVALID` | 401 | token 格式错误或已失效 |
| `SERVICE_NOT_FOUND` | 404 | 服务不存在 |
| `SERVICE_EXISTS` | 409 | 创建服务时名称已存在 |
| `SERVICE_RUNNING` | 409 | 服务运行中拒绝冒犯操作（如 stopping 时拒绝 start） |
| `SERVICE_BUSY` | 503 | 长轮询并发超限 |
| `DEPENDENCY_CYCLE` | 422 | 依赖图检测到环 |
| `DEPENDENCY_MISSING` | 200 | 依赖服务不存在（仅警告，不拒绝） |
| `SERVICE_CONFIG_INVALID` | 422 | service.yaml 校验失败 |
| `RUNTIME_NOT_FOUND` | 422 | runtime 别名无法解析 |
| `RUNTIME_NOT_EXECUTABLE` | 422 | runtime 路径不可执行 |
| `RUNTIME_USER_NOT_FOUND` | 422 | run_as 指定的用户不存在 |
| `EXTENSION_NOT_FOUND` | 404 | 扩展不存在 |
| `EXTENSION_FAILED` | 200 | 扩展运行失败（已记录，不报错给调用方） |
| `RUN_NOT_FOUND` | 404 | 任务不存在 |
| `RUN_ALREADY_DONE` | 409 | 取消已完成的任务 |
| `FILE_NOT_FOUND` | 404 | 文件不存在 |
| `FILE_PERMISSION` | 403 | 文件权限不足 |
| `FILE_TOO_LARGE` | 413 | 上传超过 max_upload_size_mb |
| `FILE_ACCESS_DENIED` | 403 | 路径不在白名单内 |
| `INVALID_REQUEST` | 400 | 请求参数错误 |
| `INTERNAL_ERROR` | 500 | 服务端内部错误 |

## 第六部分：实施细节确认

> 以下细节均已确认，开发者直接按此实施。

### 6.1 已确认的实施细节

1. **全局搜索**：搜服务名、扩展名、文件名、事件内容，**不做模糊匹配**，仅精确匹配
2. **服务/扩展禁用后**：仍出现在列表中，灰显或标注"已禁用"
3. **多标签编辑器标签持久化**：用 localStorage
4. **前端错误处理**：用 React Error Boundary 包裹主要页面组件，崩溃时显示"页面出错，请刷新"+ 错误摘要（错误类型 + 简短信息），提供"刷新页面"按钮；不做复杂错误上报
5. **扩展试运行日志**：仅在前端保存，刷新页面即丢失，后端不存储
6. **配置文件损坏时**：单个配置文件 YAML 解析失败 → 标记该服务/扩展为"配置错误"状态，不应用新配置，**不影响其他服务和 supd 启动**；提示错误位置（YAML 行号 + 字段名）
7. **PID 复用检测**：不做额外检测，依赖系统机制（物理机 systemd 回收孤儿，容器整体重启）
8. **cron 表达式时区**：用 supd 所在时区（即 Asia/Shanghai），不支持每扩展配置
9. **事件流容量上限**：内存环形缓冲 **200 条**，超出丢最旧；落盘文件按日志轮转策略
10. **扩展运行日志字符编码**：UTF-8 假设，非 UTF-8 内容按字节流展示（可能乱码但不报错）
11. **服务 user 字段权限校验**：非 root 启动 supd 时，若服务配置了 user 字段且不是当前用户，拒绝启动该服务，提示具体服务名和配置位置（service.yaml 行号）
12. **演示模板**：已删除，不实现演示模板功能；首次启动 services/ 为空时直接显示空 Dashboard + 引导提示"点击新建服务开始"

### 6.2 实施时如遇本文档未覆盖的场景

开发者（含 AI）应按以下顺序处理：

1. 优先参考本文档设计原则（简单克制、按需采集、不持久化等）
2. 选择与已有设计最一致的最简实现（不引入新抽象、新框架、新中间件）
3. 在代码注释中标注"本文档未覆盖，按 X 原则实现"
4. 不擅自引入新依赖或新设计模式

**禁止**：

- 禁止"为了未来扩展"提前实现功能
- 禁止"为了灵活性"引入配置项或抽象层
- 禁止遇未覆盖场景时"暂停并提问"（用户目标是全过程不参与），应按上述规则自主决策

## 第八部分：边界场景处理

### 8.1 并发控制

- **同时编辑**：文件保存用原子写（临时文件 + rename），后写覆盖先写（不实现乐观锁 / 版本号冲突检测）
- **同时启停**：服务 stopping 时拒绝 start 请求返回 409 + SERVICE_RUNNING；服务 starting 时 start 请求返回 409 + SERVICE_RUNNING（不重复启动）
- **同服务 on_failure 串行**：同一服务的 on_failure 扩展串行执行；不同服务的 on_failure 扩展可并行执行（各自独立）

### 8.2 时序处理

- 所有时间戳用 ISO8601 带时区（+08:00）
- 长轮询的 since 游标基于单调时钟（monotonic clock）而非墙钟，避免 NTP 跳变影响
- 内部存储用单调时钟的纳秒，返回给前端时转为 ISO8601 带时区
- cron 表达式按服务器时区（Asia/Shanghai）执行，WebUI 在定时任务页面显示"按服务器时区执行"提示

### 8.3 数据完整性

- **rename 失败**：清理临时文件并返回 500 + INTERNAL_ERROR
- **vim / nano 等编辑器的 truncate+write 场景**：文档建议用户配置 vim 的 `set backupcopy=yes`（或使用原子写编辑器如 supd 自带的 Monaco）；supd 不主动适配这些编辑器
- **配置损坏**：YAML 解析失败时，保留旧配置在内存，标记该服务 / 扩展为"配置错误"状态，不应用新配置，不影响其他服务

### 8.4 启动与关机边界

- **启动中收到信号**：启动中（服务拉起过程中）收到 SIGTERM/SIGINT 时，立即停止拉起剩余服务，已启动的服务按正常关机流程停止
- **关机超时**：关机超时（shutdown_grace_seconds 到期）强制退出前，supd_lifecycle:pre_shutdown 已执行过的扩展无法取消，记录警告日志并强制 SIGKILL 所有剩余进程
- **PID 1 信号注册**：PID 1 模式显式注册 SIGTERM/SIGINT（见 2.1.9）

### 8.5 权限与身份

- **supd 启动时检测有效用户**：非 root 时记录警告并列出受限功能（不能切换用户、不能绑定 1024 以下端口、不能修改文件权限）
- **服务配置 user 字段检测**：非 root 启动时，服务配置了 user 字段且非当前用户，在配置加载时检测并拒绝启动该服务，提示具体服务名和配置位置（service.yaml 行号）
- **用户切换失败**：返回 422 + RUNTIME_USER_NOT_FOUND
- **文件权限**：服务 data/ 目录由 supd 创建，权限 0700，所有者为服务的 user（默认为 supd 用户）；日志文件由 supd 创建，权限 0644，所有者为 supd 用户（服务进程以其 user 身份写日志时，supd 提前 chmod 确保可写）

## 附录 D：术语表

> 本术语表统一文档中关键概念的定义，便于 LLM 在实现时保持理解一致。

| 术语 | 定义 |
|---|---|
| **supd** | 本项目的单二进制程序，作为 PID 1 或系统服务管理 NAS 上的进程 |
| **服务（Service）** | 一个被监督的 longrun 守护进程，由 service.yaml 定义。仅支持 longrun，不支持 oneshot |
| **扩展（Extension）** | 一段可触发执行的脚本，统一抽象 hook、cron、手动触发。是唯一的脚本执行单元 |
| **Action** | 扩展的一个执行入口，对应前端一个按钮，通过 CLI args 区分不同 action |
| **触发器（Trigger）** | 扩展被触发的方式，固定四种：on_demand / on_schedule / service_lifecycle / supd_lifecycle |
| **任务（Run）** | 扩展的一次具体执行，有唯一 run_id。任务状态固定七种：pending / running / success / failed / timeout / canceled / killed |
| **Phase** | lifecycle 触发器的阶段。service_lifecycle 有 4 个阶段：pre_start / post_ready / on_failure / pre_stop；supd_lifecycle 有 3 个阶段：pre_start / post_ready / pre_shutdown |
| **Readiness** | 服务从"进程已启"到"可提供服务"的过渡状态。检查类型固定四种：fd_notify / tcp_check / http_check / script |
| **运行时（Runtime）** | 执行服务命令或扩展脚本的解释器/二进制。三层来源：内置默认 + config.yaml 配置 + runtimes/ 目录扫描 |
| **Entry** | 扩展的入口文件（如 run.sh），相对扩展目录的路径。等同服务 command[0] 的角色 |
| **环境变量层级** | 合并顺序：全局 env → 全局扩展私有 env → 服务 env → 服务级扩展私有 env，后者覆盖前者同名变量 |
| **执行用户（run_as）** | 扩展执行时的系统用户身份。全局扩展默认 supd 用户，服务级扩展默认服务 user，可被 meta.yaml 的 run_as 字段覆盖 |
| **触发者标识（SUPD_TRIGGER_USER）** | 记录"谁触发了本次运行"，仅用于日志和审计展示，与执行用户无关 |
| **状态机** | 服务生命周期的 7 状态模型：pending / starting / up / ready / stopping / down / failed |
| **up 状态** | 进程在跑但未通过 readiness 检查的中间态。无 readiness 配置时 up 为终态 |
| **ready 状态** | 通过 readiness 检查的终态，表示服务可接收请求 |
| **failed 状态** | 永久失败，不再自动重启（达到 max_retries 或 readiness 超时） |
| **零状态启动** | supd 重启后所有服务按 autostart 字段重置，不持久化运行时状态 |
| **stdout 协议** | 扩展通过 stdout 输出约定格式行（`::progress::` / `::result::`）上报进度和结果 |
| **长轮询** | HTTP 请求挂起最多 30 秒，有新数据立即返回，无数据超时返回空。supd 唯一的实时数据推送方式 |
| **单调时钟（monotonic clock）** | 不受 NTP 跳变影响的时钟，用于长轮询 since 游标内部存储 |
| **script_tmp** | 扩展执行时的统一临时工作目录，程序不自动清理，由用户自主处理 |
| **autostart** | service.yaml 字段，控制 supd 启动时是否自动拉起该服务 |
| **concurrency** | 扩展的并发策略，固定四种：replace / serialize / parallel / debounce:Ns |
| **package** | service.yaml 的打包配置段，控制导出时包含/排除哪些文件 |

## 附录A：实现决策备忘（2026-07-07 ~ 2026-07-09）

> 本附录记录初始开发阶段的关键实现决策，供维护阶段参考。

### A.1 任务历史清理机制（REQ-F-020）

需求规格书规定"任务历史仅存内存：保留7天"，但未指定清理触发机制。
**决策**：采用惰性清理——每次记录新任务时顺带删除超期记录，无需额外 goroutine。
**实现位置**：`internal/extension/task_history.go`

### A.2 试运行API参数（REQ-F-037）

`POST /api/extensions/<name>/run` 原始请求体仅 `{"action": "<id>"}` 不足以支持试运行。
**决策**：扩展请求体，增加 `dry_run`、`env_overrides`、`timeout_override` 字段，而非新增独立端点（语义一致，避免端点膨胀）。
**实现位置**：`internal/api/extension_handler.go`

### A.3 扩展触发器 trigger_type 展示优先级（REQ-2.2.14）

扩展可配置多种触发器，但列表卡片只展示单个 `trigger_type` 字段。
**决策**：按优先级取主要触发器：`on_schedule > service_lifecycle > supd_lifecycle > on_demand`。
**实现位置**：`internal/extension/display.go`（`GetTriggerType`）

### A.4 已确认为"后续版本"的功能项

以下功能经人工裁决推迟至后续版本，当前代码库中**不应实现**：
- 实时日志跟踪（CLI follow 模式）
- fork 后台运行模式
- 配置预览 diff 信息
- `SIGTERM → 5s → SIGKILL` 扩展硬终止序列
